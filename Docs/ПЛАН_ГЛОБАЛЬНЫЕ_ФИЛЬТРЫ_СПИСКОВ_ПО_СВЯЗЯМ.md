# План: глобальные взаимные фильтры списков по связям через `CONTEST_CODE`

Документ фиксирует **понимание задачи**, **логику связей**, **состав фильтров**, **правила пересечения и взаимной подстройки вариантов**, **особенности данных** и **пошаговый план внедрения** в код.

**Статус реализации (0.2.42):** первый проход в коде — модуль **`src/global_sheet_filters.py`**, интеграция в **`sheet_list`** (`app.py`), UI в **`sheet_list.html`** и стили **`app.css`**; query-параметры **`gf_*`** заменяют прежние **`contest_type`**, **`reward_type`**, **`season`**, **`calc_mode`**.

Опора на текущую архитектуру проекта:
- списки: маршрут `GET /sheet/{code}` в `src/app.py` (`sheet_list`);
- подписи и вспомогательные индексы: `src/sheet_list_display.py` (`build_lookup_tables`); фильтрация списков — **`src/global_sheet_filters.py`**;
- хранение: отдельная таблица SQLite на лист, актуальные строки `is_current = 1`, JSON в TEXT + денормализация `j__<JSON_колонка>__<путь>` в `src/sheet_storage.py`;
- логические связи: `config.json` → `database_model.logical_relationships_ru`.

## 1. Как понята задача (кратко)

1. На **каждой** странице списка любого листа (`CONTEST-DATA`, `GROUP`, `INDICATOR`, `REWARD`, `REWARD-LINK`, `TOURNAMENT-SCHEDULE`) показывать **один и тот же набор** фильтров по полям, перечисленным ниже — даже если на самом листе этого поля **нет** (например, фильтр `CONTEST_TYPE` на листе `REWARD`).
2. Фильтрация работает **через конкурс**: по выбранным условиям находим множество **`CONTEST_CODE`**, для которых условия выполняются (с учётом того, на каком листе «живёт» поле). Затем на **текущем** листе показываем только те строки, которые **связаны** с этим множеством конкурсов (см. раздел 3).
3. Фильтры **взаимные (faceted)**: при уже выбранных значениях в части фильтров списки допустимых значений в **остальных** фильтрах **сужаются** так, чтобы не предлагать комбинации, которые дадут пустой результат (см. раздел 5).

## 2. Узел связи: `CONTEST_CODE`

Центральный идентификатор конкурса — **`CONTEST_CODE`** в листе **`CONTEST-DATA`** (справочник конкурсов).

| Лист | Как получить `CONTEST_CODE` для строки | Примечание |
|------|----------------------------------------|------------|
| **CONTEST-DATA** | Колонка `CONTEST_CODE` в самой строке | Одна актуальная строка на конкурс (ожидаемая модель; дубликаты — предмет `consistency.py`). |
| **GROUP** | Колонка `CONTEST_CODE` | В UI список **агрегирован по конкурсу** (одна строка на `CONTEST_CODE`); в БД может быть много строк GROUP на один конкурс. |
| **INDICATOR** | Колонка `CONTEST_CODE` | |
| **REWARD-LINK** | Колонка `CONTEST_CODE` | |
| **TOURNAMENT-SCHEDULE** | Колонка `CONTEST_CODE` | |
| **REWARD** | **Нет** колонки `CONTEST_CODE` | Связь только транзитивно: `REWARD_CODE` → один или несколько `CONTEST_CODE` через строки **REWARD-LINK** (`reward_links_by_reward` в `build_lookup_tables`). При противоречивых данных правила целостности уже описаны в `database_model`; для фильтра логично трактовать: строка REWARD попадает в выборку, если **существует** связь с хотя бы одним конкурсом из текущего множества. |

Итого: любой фильтр по полям «чужого» листа сводится к подмножеству **`CONTEST_CODE`**, после чего текущий лист отфильтровывается по правилам раздела 3.

## 3. Состав фильтров и откуда брать значение

Ниже — **источник истины** для определения «какой лист / какая колонка / какой путь в JSON». В скобках — уточнения к формулировке из постановки.

### 3.1. Из **CONTEST-DATA**

| Параметр | Колонка / путь | Тип в данных |
|----------|----------------|--------------|
| Тип конкурса | `CONTEST_TYPE` | Плоская TEXT |
| Бизнес-направление | `PRODUCT_GROUP` | Плоская |
| Продукт | `PRODUCT` | Плоская |

| Параметр | Колонка |
|----------|---------|
| Код показателя | `INDICATOR_CODE` |

### 3.5. Из **TOURNAMENT-SCHEDULE** (в конфиге лист: `TOURNAMENT-SCHEDULE`, файл SCHEDULE)

|----------|----------------|
| Тип периода | `PERIOD_TYPE` (плоская) |
| Статус турнира | `TOURNAMENT_STATUS` (плоская) |
| Тип расчёта | `CALC_TYPE` (плоская; в проекте уже есть фильтр «авто/ручной» через `CALC_TYPE == "0"` — новый фильтр может сосуществовать или быть объединён с существующим — см. раздел 8) |
| Сезон | Колонка **`TARGET_TYPE`** (JSON), внутри ключ **`seasonCode`** | Уже используется функция `target_type_season_code()` в `sheet_list_display.py`. |

Семантика для фильтров по расписанию: конкурс попадает в множество, если **существует** хотя бы одна строка расписания этого конкурса, удовлетворяющая выбранным условиям (пересечение условий внутри блока SCHEDULE — по согласованию: обычно **И** между выбранными параметрами внутри одного «блока» расписания).

## 4. Алгоритм: от фильтров к строкам текущего листа

Обозначения:
- \(F\) — набор активных фильтров (каждый фильтр — множество выбранных значений, как сейчас чипы; **пустой** выбор = «не ограничивает»).
- \(C(F)\) — множество `CONTEST_CODE`, удовлетворяющих **всем** активным фильтрам одновременно (**И** между разными параметрами).

### 4.1. Вычисление \(C(F)\)

Для каждого включённого фильтра строится множество конкурсов \(C_i\), на которых выполняется этот фильтр (см. ниже). Затем:

\[
C(F) = \bigcap_{i \in \text{активные фильтры}} C_i
\]

**Правило для одного фильтра с несколькими выбранными значениями** (например, два типа конкурса):  
\(C_i = \bigcup_{v \in \text{выбранные } v} C_{i,v}\) — то есть **ИЛИ** внутри одного параметра, **И** между разными параметрами.

#### Откуда берётся \(C_{i,v}\) по типу источника

1. **Поля только в CONTEST-DATA** (`CONTEST_TYPE`, `PRODUCT_GROUP`, …, плюс условия по JSON `CONTEST_FEATURE`):  
   \(C_{i,v} = \{\text{cc} \mid \text{строка CONTEST-DATA с этим cc и is_current=1 удовлетворяет условию}\}\).

2. **GROUP.GROUP_CODE**:  
   \(C_{i,v} = \{\text{cc} \mid \exists\text{ строка GROUP(cc, ...) с GROUP_code в выборе}\}\).

3. **REWARD.REWARD_TYPE** и пути в **REWARD_ADD_DATA**:  
   \(C_{i,v} = \{\text{cc} \mid \exists\text{ REWARD-LINK(cc, reward_code)} \land \text{строка REWARD(reward_code) удовлетворяет условию по полю/JSON}\}\).  
   То есть условие проверяется на **справочнике награды**, связь с конкурсом — через **REWARD-LINK**.

4. **INDICATOR.INDICATOR_CODE**:  
   \(C_{i,v} = \{\text{cc} \mid \exists\text{ строка INDICATOR с CONTEST_CODE=cc и INDICATOR_CODE в выборе}\}\).

5. **TOURNAMENT-SCHEDULE** (PERIOD_TYPE, TOURNAMENT_STATUS, CALC_TYPE, seasonCode):  
   \(C_{i,v} = \{\text{cc} \mid \exists\text{ строка расписания с CONTEST_CODE=cc и полями/JSON, попадающими под выбор}\}\).

Если **нет ни одного активного фильтра**, полагаем \(C(F)\) = все конкурсы, для которых в `CONTEST-DATA` есть актуальная строка (или не ограничиваем — эквивалентно для отображения полного списка).

### 4.2. Фильтрация строк **текущего** листа по \(C(F)\)

| Текущий лист | Строка проходит, если |
|--------------|------------------------|
| **CONTEST-DATA** | `CONTEST_CODE ∈ C(F)` (при отсутствии фильтров — без доп. ограничения). |
| **GROUP** (агрегат по конкурсу) | `CONTEST_CODE ∈ C(F)`. |
| **INDICATOR** | `CONTEST_CODE ∈ C(F)`. |
| **REWARD-LINK** | `CONTEST_CODE ∈ C(F)`. |
| **TOURNAMENT-SCHEDULE** | `CONTEST_CODE ∈ C(F)`. |
| **REWARD** | Существует связь: \(\exists\) запись в REWARD-LINK с `REWARD_CODE` строки и `CONTEST_CODE ∈ C(F)` **или** (если награда без связей) — политика: **рекомендуется** не показывать при активных «конкурсных» фильтрах или показывать только если множество связанных конкурсов пересекается с \(C(F)\); зафиксировать в реализации один вариант. |

Поиск по строке `q` (как сейчас) логично применять **после** глобального фильтра: `строка ∈ C(F)` **и** `search_blob` содержит запрос.

## 5. Взаимные фильтры (сужение вариантов)

### 5.1. Цель UX

При выбранных значениях в фильтре A выпадающий/чиповый список для фильтра B должен содержать **только** те значения, для которых существует хотя бы один конкурс, совместимый с A **и** с остальными уже выбранными фильтрами (кроме, возможно, самого B — см. ниже).

### 5.2. Рекомендуемое правило построения опций для фильтра X

**Вариант A (рекомендуемый, без «тупиков»):**  
для построения списка значений параметра **X** применяются **все активные фильтры, кроме фильтра по X** (т.е. вычисляют \(C(F \setminus X)\), затем собирают множество значений X, встречающихся у конкурсов из \(C(F \setminus X)\) на соответствующем источнике).

Так пользователь не теряет опции из‑за частично выбранного же параметра (классическая схема faceted search).

**Вариант B (строже):** опции для X считаются по \(C(F)\) включая текущий выбор по X — тогда список может схлопнуться до уже выбранного; обычно хуже для UX.

В плане реализации заложить **вариант A** по умолчанию; вариант B можно оставить флагом отладки.

### 5.3. Порядок не важен

Пересечение множеств коммутативно: итоговое \(C(F)\) не зависит от порядка применения фильтров. Взаимность обеспечивается пересчётом опций при каждом GET (или при AJAX-запросе за facet-ами).

### 5.4. Производительность

На объёме промышленных CSV (сотни строк на лист) допустимо на каждый запрос списка:
- один проход для построения «витрины» конкурсов (как расширение `build_lookup_tables`), плюс
- несколько проходов для facet-ов (в памяти), **либо**
- материализованное представление / временная таблица в SQLite (если объём вырастет).

Сейчас `build_lookup_tables` уже читает все строки ключевых листов в память — логично **расширить** эту структуру, а не плодить N отдельных полных сканов без нужды.

## 6. Мелкие детали по JSON и типам значений

1. **Денормализация `j__*`** (`sheet_storage.flat_map_for_json_column`): для скалярных листьев фильтрация в SQL может быть быстрой (`WHERE j__REWARD_ADD_DATA__getCondition__employeeRating__seasonCode IN (...)`). Для путей, которые в денормализации попали как **сериализованный JSON массива** (например, ключ целиком указывает на массив), условие «значение в массиве» может потребовать **разбора JSON** в Python при построении \(C_i\) — заложить в задачу явную проверку по каждому перечисленному ключу.
2. **`businessBlock` / `seasonItem` / `hiddenRewardList`**: заранее определить семантику фильтра:
   - **«конкурс попадает, если в JSON значение равно v»** (скаляр),
   - **«если массив — достаточно вхождения v как элемента»** (типично для списков кодов),
   - **«если объект — …»** (если встретится).
3. **Булевы / Y-N**: нормализовать сравнение с учётом того, как значения лежат в CSV/JSON (`"Y"/"N"`, `true/false`, `1/0`).
4. **`vid`**: скаляр в `CONTEST_FEATURE`; убедиться, что путь в денормализации совпадает с реальными ключами в данных.
5. **Два разных `TARGET_TYPE`**: в фильтрах из постановки — плоское поле конкурса и JSON расписания; в UI подписать явно, например «Цель расчёта (конкурс)» vs «Сезон турнира (расписание)», чтобы не путать пользователя.

## 10. Критерии приёмки

1. На **каждом** листе со списком отображается **полный** набор фильтров из раздела 3 (с понятными подписями, без дублирования имён колонок без контекста).
2. При выборе только `CONTEST_TYPE` на листе **REWARD** отображаются только награды, у которых есть связь с конкурсом выбранного типа (через REWARD-LINK).
3. При комбинации фильтров результат **не шире**, чем пересечение условий по конкурсам.
4. При изменении любого фильтра **остальные** списки значений **не содержат** вариантов, дающих пустое пересечение (при использовании правила 5.2 вариант A).
5. Пустой выбор по параметру не отфильтровывает строки по этому параметру.
6. Поиск `q` и глобальные фильтры работают **совместно** (И).
