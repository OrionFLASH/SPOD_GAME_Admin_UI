# Предложения по `label`/`description` для параметров геймификации

Источник анализа: экспорт документов в `Docs/ПКАП параметры/(GAME) [HDP] Справочники по геймификации` (начиная с `index.html`, далее страницы `dm_gamification_*`).
Дополнительно сверено с `Docs/ПКАП параметры/contestscheme_v0.json` (JSON-схема конструктора конкурса).

Цель: подготовить список правок для программы (в первую очередь для `editor_field_ui` в `config.json`), чтобы у каждого параметра был:
- короткий, понятный `label`;
- краткий `description` с бизнес-смыслом.

Изначально файл создавался как список предложений без изменения кода. Часть предложений уже применена в `config.json` (блок `editor_field_ui`), статус отмечен ниже.

## Уточнение по связям и соответствиям полей

- `CONTEST_CODE` — основной ключ связей между листами `CONTEST-DATA`, `GROUP`, `INDICATOR`, `TOURNAMENT-SCHEDULE`, `REWARD-LINK`.
- `REWARD` связывается с конкурсом через `REWARD-LINK` (`CONTEST_CODE` + `GROUP_CODE` + `REWARD_CODE`), а не напрямую.
- В `contestscheme_v0.json` вложенные блоки соответствуют витринам:
  - `contest_group[*]` ↔ `dm_gamification_contest_group`
  - `contest_indicator[*]` ↔ `dm_gamification_contest_indicator`
  - `tournament_schedule[*]` ↔ `dm_gamification_tournament_schedule`
- В схеме конструктора есть имена, отличающиеся от CSV:
  - `contest_full_name` ↔ `FULL_NAME`
  - `tournament_period` ↔ `FILTER_PERIOD_ARR`
  - `contest_group[*].reward_code` — в актуальной витрине вынесено в отдельный лист `REWARD-LINK`.
- Для методов расчёта и постобработки (`plan_method_code`, `contest_factor_method`, `indicator_add_calc_type` и т.д.) источником допустимых значений считать `dm_gamification_method_dic`.

## 1) Лист `CONTEST-DATA` (`dm_gamification_contest_data`)

Источник: `(GAME)_[HDP]_dm_gamification_contest_data.html`.

| Параметр | Предлагаемый label | Предлагаемый description |
|---|---|---|
| `CONTEST_CODE` | Код конкурса | Уникальный идентификатор конкурса в системе. |
| `FULL_NAME` | Название конкурса | Человекочитаемое наименование конкурса. |
| `CREATE_DT` | Дата создания | Дата создания записи конкурса. |
| `CLOSE_DT` | Дата завершения | Плановая или фактическая дата завершения конкурса. |
| `BUSINESS_STATUS` | Бизнес-статус | Текущий статус конкурса (например, активный/закрыт). |
| `CONTEST_TYPE` | Тип конкурса | Категория конкурса (турнирный и т.д.). |
| `CONTEST_DESCRIPTION` | Описание конкурса | Краткое текстовое описание цели и правил конкурса. |
| `CONTEST_FEATURE` | Параметры конкурса (JSON) | Дополнительные настройки конкурса в JSON-структуре. |
| `SHOW_INDICATOR` | Показатель на витрине | Показатель, который отображается как ключевой в UI/отчётности. |
| `PRODUCT_GROUP` | Бизнес-направление | Группа/направление продукта для конкурса. |
| `PRODUCT` | Продукт | Продукт, к которому относится конкурс. |
| `FACTOR_MARK_TYPE` | Тип оценки | Способ оценки результата (например, рейтинг). |
| `FACTOR_MATCH` | Правило сравнения | Как определяется лучший результат (например, MAX/MIN). |
| `CONTEST_SUBJECT` | Объект конкурса | Уровень расчёта конкурса: сотрудник или подразделение. |
| `TARGET_TYPE` | Вид цели расчёта | Тип цели (например, ПРОМ/ИССЛ). |
| `SOURCE_UPD_FREQUENCY` | Частота обновления (дни) | Периодичность обновления исходных данных для расчёта. |
| `PLAN_METHOD_CODE` | Метод расчёта плана | Способ определения планового значения показателя. |
| `PLAN_MOD_METOD` | Операция модификации плана | Операция для перерасчёта плана (умножение/добавление). |
| `PLAN_MOD_VALUE` | Значение модификатора плана | Число, применяемое в выбранном методе модификации плана. |
| `CONTEST_INDICATOR_METHOD` | Метод расчёта индикатора | Алгоритм формирования индикатора конкурса (интегральный/отношение). |
| `CONTEST_FACTOR_METHOD` | Метод итогового показателя | Правило расчёта итогового значения конкурса по периодам. |
| `CONTEST_PERIOD` | Правила периодов (JSON) | Настройки периодов и порогов для оценки выполнения конкурса. |
| `BUSINESS_BLOCK` | Бизнес-блок | Перечень бизнес-блоков, участвующих в конкурсе. |
| `FACT_POST_PROCESSING` | Постобработка факта | Дополнительная логика обработки рассчитанного фактического показателя. |
| `CALC_TYPE` | Тип расчёта | Режим выполнения расчёта (промышленный/ручной/версионный). |

Примечание: `LOAD_DT`, `LOAD_TS` лучше оставить как технические поля с минимальными пояснениями или скрывать в пользовательском сценарии.

## 7) Справочники-источники для выпадающих списков

### `dm_gamification_method_dic`
Источник: `(GAME)_[HDP]_dm_gamification_method_dic.html`.

Поля для `label`/`description`:
- `ELEMENT_CODE` → `Код атрибута`
- `ELEMENT_VALUE` → `Значение атрибута`
- `ELEMENT_DESCRIPTION` → `Описание значения`
- `ISACTIVE_FLG` → `Активно`

Этот справочник полезен для человекочитаемых подсказок к значениям:
- `contest_data.plan_method_code`
- `contest_data.plan_mod_metod`
- `contest_data.contest_indicator_method`
- `contest_data.contest_factor_method`
- `contest_indicator.indicator_add_calc_type`
- `contest_indicator.indicator_agg_function`

### `pre_dm_gamification_source_indicator_dic`
Источник: `(GAME)_[HDP]_pre_dm_gamification_source_indicator_dic.html`.

Поля для `label`/`description`:
- `SOURCE_CODE` → `Код источника`
- `INDICATOR` → `Индикаторы источника (JSON)`
- `FILTERED_ATTRIBUTE` → `Доступные фильтры (JSON)`
- `STATUS` → `Статус источника`

Рекомендация: использовать как базу подсказок для `INDICATOR_FILTER` и `CONTEST_CRITERION`.

## 9) Техническое примечание для последующего шага

Для внедрения в UI панели эти формулировки логично вносить в:
- `config.json` → блок `editor_field_ui` (`label`, `description`, `show_description`);
- при необходимости в `field_enums` (`label/value`) для более понятных названий в выпадающих списках/переключателях.
