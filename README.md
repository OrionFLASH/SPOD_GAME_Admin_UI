Локальная панель администратора для просмотра и правки выгрузок турниров SPOD (CSV с разделителем `;`), с хранением в SQLite и упрощёнными проверками связей между листами. Запуск одной командой `python main.py` (встроенный Uvicorn), без Docker и отдельных сервисов.

**Репозиторий:** [SPOD_GAME_Admin_UI](https://github.com/OrionFLASH/SPOD_GAME_Admin_UI) — код панели в **корне** этого каталога (ранее каталог `spod_tournament_admin/` в репозитории SPOD_PROM). История коммитов, относящаяся к панели, перенесена из старого репозитория командой `git subtree split -P spod_tournament_admin` (сообщения и содержимое коммитов сохранены).

**Локальная разработка:** откройте в Cursor корень этого репозитория (`SPOD_GAME_Admin_UI`), чтобы подсказки и поиск кода относились только к панели.

**История агента Cursor (локально):** в каталог данных Cursor для этого workspace скопированы архивы диалогов из старых папок проекта (`SPOD-PROM` и при открытии подпапки — `spod-tournament-admin`):  
`~/.cursor/projects/Users-orionflash-Documents-MyProject-SPOD-GAME-Admin-UI/agent-transcripts/`  
(файлы `*.jsonl` в подкаталогах по UUID). После копирования имеет смысл **полностью закрыть Cursor и открыть проект заново** — отображение старых чатов в боковой панели зависит от версии Cursor и не документировано как официальная функция; при отсутствии чатов в UI файлы всё равно можно открыть вручную для справки. В Git эти файлы не входят.

**Пути:** `IN/`, `OUT/`, `log/` — относительно корня этого репозитория. Свой `requirements.txt` и виртуальное окружение `.venv/` в корне.

**`.gitignore`:** в репозитории не коммитятся `OUT/`, логи, `.venv/`; входные CSV в `IN/**/*.csv` **разрешены** к коммиту (см. исключение в `.gitignore`).

## 1. Формулировка задачи и техническое задание

1. Импорт шести листов из `IN/SPOD` в базу данных SQLite.
2. Просмотр списка строк по каждому листу с поиском (включая подтянутые названия), колонками «Ключ», «Название», при необходимости «Связи» и при **`consistency_column.show`** в **`config.json`** — колонка консистентности (бейджи OK / «Проблемы» по **`consistency_ok`** в БД). Для **CONTEST-DATA** и **GROUP** в репозитории колонка консистентности в таблице списка **скрыта** (**`show`: false**); для **INDICATOR**, **REWARD**, **REWARD-LINK**, **TOURNAMENT-SCHEDULE** — **показана**. (**CONTEST-DATA:** «Код конкурса» (расширенная колонка), «Наименование», «Тип конкурса» `CONTEST_TYPE`, «Просмотр»; колонка «Связи» в таблице **не** выводится — турниры и награды остаются в **`search_blob`** для поиска по `q`; фильтр **`contest_type`** — множественный выбор **чипами-чекбоксами**, как **`reward_type`** / **`season`** на других листах; длинный текст в ячейках переносится — **`sheet_list.html`**, **`app.css`**; порядок и подписи колонок данных (после **#**) — **`config.json` → `sheet_list_columns`** (**разделы 6c.6**, **6c.7**)); лист **GROUP** в списке — **одна строка на конкурс** (код и название из CONTEST-DATA, в «Связях» перечень уровней **GROUP_CODE : GROUP_VALUE**); «Просмотр» открывает карточку со всеми уровнями этого конкурса и отдельным сохранением по каждой строке GROUP; для листа **REWARD** — колонки «Награда» (`FULL_NAME` награды), «Название / описание» (**название конкурса** из CONTEST-DATA по цепочке REWARD-LINK → `CONTEST_CODE`), **GROUP_CODE** из связей, фильтр по **REWARD_TYPE** и строка **«Конкурс:»** в «Связи» по **REWARD-LINK**; для листа **REWARD-LINK** — **четыре** колонки данных (колонка «Связи» скрыта): **«Код награды»** (`REWARD_CODE`), **«Уровень»** (`GROUP_CODE`), **«Название награды»** и **«Название турнира»** (соответственно `FULL_NAME` награды из REWARD и конкурса из CONTEST-DATA по справочникам); **`CONTEST_CODE`** в таблице не выводится, но участвует в полнотекстовом **поиске** вместе с JSON ячеек строки; оформление таблицы — класс **`.data-list-reward-link`** в **`app.css`** (фиксированная раскладка: узкие колонки кодов, расширяемые — названия).
3. Просмотр и редактирование одной строки: плоские поля в **сетке** с поиском; для части плоских колонок и **отдельных листьев внутри JSON-колонок** — **настраиваемые числовые форматы** (целое / дробное, диапазон, условные ветки по другому полю) из `config.json`: в **`editor_field_numeric`** задаётся **`column`** (имя JSON-колонки или плоской колонки) и при необходимости **`json_path`** до листа (как в **`field_enums`**); при совпадении пути числовое правило **имеет приоритет** над **`field_enums`** для этого листа. Примеры: лист **INDICATOR** — **`INDICATOR_VALUE`** (**0…1 000 000 000**), **`INDICATOR_WEIGHT`** (**−10…1000**, пустая ячейка); лист **REWARD** — поля **`REWARD_ADD_DATA`** (`itemMinShow`, лимиты рейтинга и т.д.); лист **TOURNAMENT-SCHEDULE** — **`CRITERION_MARK_VALUE`** и **`FILTER_PERIOD_ARR` → `criterion_mark_value`** (**0…1 000 000**, пусто допустимо). Дублирующие **`field_enums`** для таких полей убраны. JSON-колонки — **развёртка в отдельные параметры** (строка, число, логический, null) плюс режим **«Сырой JSON»**; боковая навигация по разделам; признак изменённых полей (предыдущее значение рядом); нижняя панель **«Сохранить в базу»** / **«Отменить правку»** (запись в БД только по сохранению; отмена — перезагрузка страницы с сервера); переходы по **связям** между строками разных листов с цепочкой возврата и подменю в колонке «Разделы». **Защита от потери правок:** при несохранённых отличиях формы от загруженных данных — перехват ухода на другой адрес, шаг «назад» по цепочке, переход по крошкам, ссылка «Назад к списку», отправка формы **«Остановить»** с показом **собственного** диалога (сохранить и уйти / выйти без сохранения / остаться); при закрытии вкладки или обновлении страницы — стандартное предупреждение браузера (`beforeunload`). Кнопки дока «Сохранить в базу» и «Отменить правку» **не** должны дублировать этот диалог; после выбора в модалке ухода не должно появляться **второго** системного окна из‑за `beforeunload`.
4. Сохранение **новой версии** строки в БД (старая версия остаётся в истории, актуальна только последняя) с пересчётом проверок; опционально запрет сохранения при ошибках (`consistency.mode: strict`).
5. Экспорт листа обратно в CSV (`;`, UTF-8).
6. Переимпорт всех CSV из папки (сброс данных в БД).
7. Светлый интерфейс в духе «liquid glass» (полупрозрачные панели, размытие фона).
8. Кнопка **«Остановить»** в шапке на всех экранах: POST `/admin/stop` — завершение процесса панели (и дочерних процессов на POSIX через `pgrep -P`).
9. **Мастер создания конкурса** по шагам (CONTEST-DATA → … → предпросмотр) с промежуточным сохранением черновика в БД, возобновлением по списку или `?draft=uuid`, финальной атомарной вставкой строк в листы и защитой ухода со страницы при несохранённом относительно сервера состоянии.

## 2. Описание решения

- **Стек:** Python 3, FastAPI, Uvicorn, Jinja2, стандартный `sqlite3`, статика и шаблоны в `src/`.
- **Конфигурация:** `config.json` — пути, порт, список листов `sheets` (импорт и UI: в т.ч. **`file`** — имя CSV в `IN/SPOD`), справочник **`sheet_bindings`**, пояснение модели БД `database_model`, перечисления **`field_enums`**, подсказки **`editor_textareas`**, **числовые форматы** **`editor_field_numeric`**, порог длинного текста **`editor_long_text_threshold`**, режим **`consistency`**. Списки листов: **`sheet_list_lookups`** (справочники для колонок), **`sheet_list_columns`** по каждому листу — **`rules`** (колонки данных, **`value`** — откуда брать значение), **`consistency_column`** (показ / подпись / класс колонки консистентности) — см. **разделы 6c.6** и **6c.7** (пошаговое добавление колонки).
- **Развёртка настроек редактора:** `src/editor_config.py` — превращает сгруппированные `field_enums` / `editor_textareas` / `editor_field_ui` / **`editor_field_numeric`** в плоские списки для `row_editor.js` и мастера.
- **Проверка конфига:** `src/config_validate.py` — при старте: **`validate_sheet_bindings`**, **`validate_field_enum_sheet_options`**, **`validate_sheet_list_lookups`**, **`validate_sheet_list_column_values`** (предупреждения в лог).
- **БД:** файл `OUT/DB/tournament_admin.sqlite`; реестр листов **`sheet`** (в т.ч. `headers_json` — порядок колонок CSV); для каждого листа — отдельная таблица **`spod_sheet_<КОД>`** (`src/sheet_storage.py`, `src/db.py`): колонки как в CSV, JSON-колонки как TEXT с полным JSON, плюс денормализованные листья **`j__<имя_JSON_колонки>__<путь>`** для SQL-запросов по отдельным полям. Версионирование: `is_current`, `replaces_row_id`, `sort_key`, `row_index`, `sheet_id` — как раньше по смыслу, но на физической таблице листа. Повторное сохранение без изменений — HTTP 400. **`wizard_draft`** не участвует в импорте CSV. При обнаружении устаревшей таблицы **`data_row`** она удаляется, данные листов нужно снова загрузить импортом (автоматически при пустом `sheet`).
- **Импорт:** `src/ingest.py` читает CSV, создаёт/пересоздаёт таблицу листа и вставляет строки с денормализацией JSON.
- **JSON SPOD:** `src/spod_json.py` — нормализация тройных кавычек и попытка `json.loads` для отображения в редакторе; на клиенте ту же логику повторяет **`SpodJsonEditor.tryParseSpodJsonCell`** в `row_editor.js` (сравнение «грязного» JSON и мастер при копии из БД).
- **Проверки:** `src/consistency.py` — множества кодов конкурсов/наград; для **GROUP** учитывается уникальность тройки **`(CONTEST_CODE, GROUP_CODE, GROUP_VALUE)`**; **REWARD-LINK** по-прежнему проверяется на существование пары **`(CONTEST_CODE, GROUP_CODE)`** среди строк GROUP; также INDICATOR, TOURNAMENT-SCHEDULE.
- **Связи на карточке строки:** `src/relations.py` — выборки связанных фрагментов по кодам; в шаблоне — ссылки на `/sheet/{code}/row/{id}` с подписью `preview` (**«: значение»**; для **GROUP** — **`GROUP_CODE : GROUP_VALUE`**; для **REWARD-LINK** — **`GROUP_CODE : REWARD_CODE`**), якоря `rel-card-*` для прокрутки к карточке.
- **Навигация по связям и возврат:** `src/static/trail_nav.js` — при переходе по связи в `sessionStorage` сохраняется цепочка страниц; со списка листа в строку добавляется `?from_list=1` (после загрузки параметр убирается из адреса), чтобы первым шагом «назад» был список этого листа; панель над формой: крошки, «Шаг назад», «К списку листа», «На главную»; сброс цепочки при открытии `/` и при ссылке «Панель турниров». Дополнительно переносится `list_qs` (query списка), поэтому возврат из карточки и переходы по связям сохраняют выбранные фильтры/поиск списка.
- **Шапка с переключением листов:** `src/templates/base.html` + `app._topbar_sheet_links` — в верхней панели на всех страницах отображаются кнопки кодов листов (`CONTEST-DATA`, `GROUP`, `INDICATOR`, `REWARD`, `REWARD-LINK`, `TOURNAMENT-SCHEDULE`) с подсветкой активного листа. При переключении сохраняется текущий query списка, чтобы не терять состояние фильтров.
- **Список листа REWARD (и родственные списки):** `sheet_list_display.py`, **`sheet_list_column_values.py`**, ветка **`sheet_list`** — вычисление полей (**`display_for_sheet_row`**), применение **`sheet_list_columns.rules[].value`**, колонка консистентности по **`consistency_column`**. Шаблон **`sheet_list.html`**: **`list_columns`**, **`consistency_column`**. Конфигурация — **`sheet_list_lookups`**, **`sheet_list_columns`** (**разделы 6c.6**, **6c.7**). Для **REWARD**: «Награда» (`reward_name_col`), **CONTEST_CODE по связям** (`reward_contest_codes_col`), «Название / описание» (`title_line` из CONTEST-DATA по REWARD-LINK), **GROUP_CODE** (`group_codes_col`), «Связи» (`relations_line`); множественный фильтр **`gf_reward_type`**. Для **REWARD-LINK**: поля **`reward_link_*`**; стили **`.data-list-reward-link`**. Для **TOURNAMENT-SCHEDULE**: в т.ч. **`TOURNAMENT_STATUS`**, **`CONTEST_CODE`**, **`FULL_NAME`** конкурса (`schedule_contest_name_col`), период, сезон из **TARGET_TYPE**, связанные награды; фильтр по сезону в **`gf_*`**. Для **CONTEST-DATA** — **`CONTEST_TYPE`**. Глобальные фильтры **`gf_*`** — **чекбоксы-чипы**; кнопка «Сбросить все фильтры». Общая форма в **`sheet_list.html`** / **`app.css`**; подробности — **раздел 6c**.
- **Остановка сервера из UI:** `src/server_stop.py` + маршрут `POST /admin/stop` — после ответа браузеру планируется `SIGTERM` дочерним процессам текущего PID (если есть), затем процессу панели; на POSIX через ~2.5 с при необходимости повторно отправляется `SIGKILL` самому процессу. **Без пароля:** рассчитано на локальный `127.0.0.1`; не выставляйте панель в открытую сеть без прокси с авторизацией.
- **Экспорт:** `src/export_csv.py` — восстановление CSV по заголовкам из первой строки данных листа.
- **Мастер «Создать конкурс»:** кнопка на главной → `GET /wizard/new-contest` (`wizard_new_contest.html`, `wizard_contest.js`); подключается **`row_editor.js`** для общего UI JSON-колонок (`window.SpodJsonEditor`), бейджей ограничений (`window.SpodFieldUiSignals`), **числовых плоских полей** (`window.SpodNumericField`, те же правила `fieldNumeric`, что на карточке строки) и **двухпозиционных перечислений** (`window.SpodYnField` — см. **раздел 4.4**). Порядок шагов: CONTEST-DATA → … → предпросмотр; **`POST /wizard/new-contest/commit`** — вставка строк в таблицы листов `spod_sheet_*`; ответ **303** с `Location` на карточку конкурса; на клиенте переход отложен (`setTimeout` + `location.assign`), чтобы не ловить ложный **AbortError** у `fetch`; ошибки разбираются из `response.text()` (JSON с `detail` или текст). **Коды:** `REWARD_CODE` в шаге REWARD-LINK — префикс `r_<CONTEST_CODE>` и поле суффикса (при нескольких связях суффикс обязателен и уникален); `TOURNAMENT_CODE` — префикс `t_<CONTEST_CODE>_` и **ровно четыре цифры**; проверки дублируются в `wizard_contest.validate_payload`. **Черновик:** `wizard_draft`, `PUT`/`GET`/`DELETE` draft API; при commit передаётся `draft_uuid` — черновик удаляется в транзакции. **Уход:** модалка и `beforeunload` при несохранённом относительно сервера состоянии.
- **Редактор строки (клиент):** `row_editor.js` — карточка строки из `#row-editor-bootstrap`; плоские поля и JSON-колонки с режимами **«По полям»** / **«Сырой JSON»**; пустой JSON и `{}` дополняются шаблоном из `field_enums` и обязательных `json_path` в `editor_field_ui` (`mergeDeclaredJsonTemplate`); `field_enums` — select и «Задать своё…», при **`input_display`** и двух опциях — **двухпозиционный переключатель** (см. **раздел 4.4**); опционально **`whitelist_validated_input`** — текстовый ввод со списком допустимых значений из **`options`** и цветовой подсветкой (см. **раздел 4.5**); элементы **`options`** могут быть строками (значение = подпись) или объектами **`{ "label": "…", "value": "…" }`** для русских подписей при неизменном значении в БД (см. **раздел 4.3**); `editor_textareas` — высота textarea, явный тип **`input_type: "text"`** для обычного многострочного текста и **поля дат** (в т.ч. с `json_path` внутри JSON-колонки); для массива **однотипных объектов** в JSON — **`json_object_array`** + **`object_array_item_keys`** (см. **раздел 4.5**); **`fieldNumeric`** (`editor_field_numeric`) — отдельные `<input>` с проверкой диапазона, для дробных — нормализация при потере фокуса (запятая как разделитель, фиксированное число знаков после запятой, дополнение нулями, округление при усечении), предупреждения при недопустимом вводе; порог длинного текста; ограничения полей — бейджи **«Обязательно»** / **«Не пусто»**; экспорт **`SpodJsonEditor`** / **`SpodFieldUiSignals`** / **`SpodNumericField`** / **`SpodYnField`** для мастера. Для **`REWARD` / `REWARD_ADD_DATA`** по **`REWARD_TYPE`** (плоское поле / мастер): тип задан — в «По полям» только ключи каталога `Docs/JSON/SPOD_INPUT_DATA_CATALOG.md`, разрешённые для этого типа (в т.ч. после смены типа блок пересобирается); тип пуст — только ключи, входящие во **все** типы из матрицы (пересечение); ключи **вне** каталога при непустых данных по-прежнему видны; шаблон merge не создаёт пути, не проходящие эти правила. **Разбор JSON в мастере:** перед развёрткой вызывается **`tryParseSpodJsonCell`** (та же нормализация SPOD, что и `spod_json.try_parse_cell`), чтобы при загрузке копии конкурса из БД работал режим «По полям» для колонок вроде `CONTEST_FEATURE`. **Защита ухода (`installLeaveGuard`):** перехват ссылок, модалка, `beforeunload`; флаг `leaveGuardSuspended`; `spod_edit_trail`; стили `.spod-leave-modal-overlay` в `app.css`.

Точка входа: `main.py` или **`run.sh`** поднимают Uvicorn на `host`/`port` из конфига (по умолчанию `http://127.0.0.1:8765/`).

<a id="readme-41"></a>
### 4.1. Логическая модель связей между листами

Полный текст в **`config.json` → `database_model.logical_relationships_ru`**. Кратко:

| Откуда | Куда | Ключ | Кардинальность | Смысл |
|--------|------|------|------------------|--------|
| **CONTEST-DATA** | — | `CONTEST_CODE` | уникален среди актуальных строк | Справочник конкурсов |
| **GROUP** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Уникальность строки GROUP — тройка **`CONTEST_CODE` + `GROUP_CODE` + `GROUP_VALUE`** |
| **REWARD-LINK** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много связей |
| **REWARD-LINK** | GROUP | `(CONTEST_CODE, GROUP_CODE)` | **N:1** | В GROUP должна быть хотя бы одна строка с этой парой (несколько строк отличаются **`GROUP_VALUE`**) |
| **REWARD-LINK** | **REWARD** | `REWARD_CODE` | **N:1** | Много строк связи могут ссылаться на **одну** строку награды с тем же кодом (в быту говорят «награда одна — ссылок много»; не путать с дублированием строк в REWARD) |
| **REWARD** | CONTEST-DATA | транзитивно через REWARD-LINK | согласование | Все `REWARD-LINK` с данным `REWARD_CODE` должны иметь **один и тот же** `CONTEST_CODE`; иначе награда «приписана» к разным конкурсам |
| **INDICATOR** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много показателей |
| **TOURNAMENT-SCHEDULE** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много строк расписания |

Проверки: `consistency.py` — ссылки на существующие коды, уникальность `CONTEST_CODE` / `REWARD_CODE` в справочниках, отсутствие «висячих» наград без `REWARD-LINK`, запрет разных `CONTEST_CODE` для одного `REWARD_CODE` в связях.

<a id="readme-42"></a>
### 4.2. Числовые поля (`editor_field_numeric`)

Назначение: задать для выбранных **плоских** колонок листа и для **отдельных числовых листьев внутри JSON-колонки** тип ввода **целое число** или **число с дробной частью**, границы `min` / `max`, а также **условные** форматы в зависимости от значения другой колонки той же строки (та же сетка формы / тот же шаг мастера). Для листа JSON в правиле указываются **`column`** = имя JSON-колонки (например **`REWARD_ADD_DATA`**, **`FILTER_PERIOD_ARR`**) и **`json_path`** до скалярного числа; клиент **`row_editor.js`** подбирает правило функцией **`findNumericRuleDef(bootstrap, column, jsonParts)`** и рендерит текстовое поле с нормализацией по blur, как у плоских числовых ячеек.

**Группировка в JSON** (как у `field_enums`): массив блоков `{ "sheet_code": "<КОД_ЛИСТА>", "rules": [ ... ] }`. Допускается устаревший плоский вид одного правила с полем `sheet_code` на каждой записи.

**Статическое правило** (один формат на колонку):

| Поле правила | Назначение |
|----------------|------------|
| `column` | Имя колонки CSV: **плоская** ячейка или **JSON-колонка**, если задан `json_path` |
| `json_path` | Опционально: массив сегментов пути внутри JSON (строки и/или числа — индексы массивов), как в **`field_enums`**. Без `json_path` правило относится только к **плоской** колонке с именем `column`. |
| `format` | `"integer"` — только целое; `"decimal"` — дробное с фиксированным числом знаков после запятой в строке ячейки |
| `min`, `max` | Допустимый числовой диапазон (включительно) |
| `decimal_places` | Только для `format: "decimal"`: сколько знаков после точки хранить и показывать после нормализации; если не задано — **5** |
| `comment` | Произвольный комментарий в конфиге (в UI не выводится) |

**Условное правило** (формат выбирается по значению другой колонки):

| Поле | Назначение |
|------|------------|
| `column` | Редактируемая колонка |
| `conditional_formats` | Массив веток. Каждая ветка: **`when`**: `{ "column": "<имя_колонки>", "equals": "<строка>" }` — сравнение с текущим значением зависимой колонки **как со строкой** (после `trim`); далее в той же ветке — **`format`**, **`min`**, **`max`**, при необходимости **`decimal_places`** |
| `default_format` | Если ни одна ветка `when` не подошла: объект с полем **`format`**. Значение **`"empty_only"`** означает: поле должно оставаться **пустым**, ввод **заблокирован** (режим «ни то ни другое» из предметной логики) |

Порядок проверки веток — порядок элементов в `conditional_formats`; срабатывает первая подходящая ветка.

**Поведение клиента** (`row_editor.js`, мастер — `wizard_contest.js` через `window.SpodNumericField`):

- Для **`integer`**: ввод нормализуется к целому, значение должно попадать в `[min, max]`; иначе показывается предупреждение.
- Для **`decimal`**: допускается ввод с **запятой** или **точкой** как разделителем дробной части; при потере фокуса строка приводится к виду **целая часть + `.` + дробная часть** длины ровно `decimal_places` (недостающие справа знаки дополняются нулями; лишние — усечение с **округлением** последнего оставшегося знака по правилам IEEE для числа с плавающей точкой в JavaScript). Примеры нормализации: `12` → `12.00000`; `13,03` → `13.03000`; при пяти знаках после запятой значение с большим числом десятичных цифр сначала округляется до нужной точности. Если после разбора не число или число вне `[min, max]` — предупреждение, сохранение по-прежнему может быть отклонено бэкендом или помечено консистентностью в зависимости от остальных правил.
- При смене зависимой колонки (например, в сетке мастера) для всех числовых полей вызывается обновление состояния (разблокировка / блокировка / смена подсказок).
- Перед сбором payload для сохранения выполняется **blur** у активных числовых полей, чтобы зафиксировать нормализацию.

**Примеры в репозитории** (`config.json`):

- **`CALC_TYPE`**: на листах **CONTEST-DATA**, **TOURNAMENT-SCHEDULE** и **INDICATOR** — **`field_enums`** с вариантами **`"0"`** и **`"1"`** (без **`editor_field_numeric`** для этой колонки); в выгрузке **INDICATOR** значения **`3`** (и единичное **`2`**) в поле приведены к **`1`**, чтобы не оставалось значений вне списка (CSV и БД при обновлении данных).
- **INDICATOR** — **`INDICATOR_VALUE`**: только **`editor_field_numeric`** (`format: "integer"`, **`min`**: 0, **`max`**: 1 000 000 000); перечисление в **`field_enums`** для этой колонки **не используется**. В **`editor_field_ui`** — подпись диапазона и **`show_description`**: true.
- **INDICATOR** — **`INDICATOR_WEIGHT`**: **`editor_field_numeric`** (`integer`, **`min`**: −10, **`max`**: 1000); пустое значение допускается (**`allows_empty`**: true в **`editor_field_ui`**; на клиенте пустая строка после blur остаётся пустой). Ранее задаваемый **`field_enums`** с коротким списком весов удалён.
- **INDICATOR** — **`SOURCE_UPD_FREQUENCY`**: только **`editor_field_numeric`** (`integer`, **`min`**: 1, **`max`**: 10); перечисление в **`field_enums`** для этой колонки не используется, поле редактируется как числовое.
- **GROUP** — колонки **`GET_CALC_CRITERION`**, **`ADD_CALC_CRITERION`**, **`ADD_CALC_CRITERION_2`**: при **`GET_CALC_METHOD`** **`"1"`** — дробное 0–1 с пятью знаками (как в выгрузке для режима 1); при **`"2"`** — целое 0–50; при **`"3"`** — ввод запрещён, поле должно оставаться **пустым** (**`format`: `empty_only`** в условной ветке); для прочих значений метода — **`default_format`: `empty_only`**. Для **`GET_CALC_CRITERION`** задано только **`editor_field_numeric`** (без **`field_enums`**). В **`row_editor.js`** для плоского поля при наличии правила **`editor_field_numeric`** перечисление **`field_enums`** для той же колонки не используется (приоритет числа). В конфиге у блока указан `comment`: в тексте ТЗ могло фигурировать имя **`GET_CALC_CRITERION`**, в выгрузке условие завязано на **`GET_CALC_METHOD`**.
- **REWARD**, колонка **`REWARD_ADD_DATA`**: для ряда листьев (`itemMinShow`, `itemLimitCount`, условия **`getCondition.employeeRating.*`**, **`minCrystalEarnedTotal`** и др.) заданы статические правила **`integer`** с диапазонами; пустая строка там, где в **`editor_field_ui`** указано **`allows_empty`**: true, после blur остаётся пустой (см. **`applyNumericFormatToValue`** в **`row_editor.js`**).
- **TOURNAMENT-SCHEDULE**: плоское **`CRITERION_MARK_VALUE`** и лист **`FILTER_PERIOD_ARR`** с путём **`[0, "criterion_mark_value"]`** — целое **0…1 000 000**; **`field_enums`** с коротким списком чисел для этих полей удалён. Лист **REWARD** — плоские **`REWARD_COST`** (0…100) и перечисленные листья **`REWARD_ADD_DATA`** в том же блоке **`editor_field_numeric`**.

Сервер при старте **не валидирует** синтаксис `editor_field_numeric` отдельным модулем (в отличие от частичной проверки `sheet_bindings`); ошибки в структуре приводят к тому, что правило может быть проигнорировано на клиенте. Рекомендуется держать конфиг валидным JSON и следовать таблице выше.

### 4.3. Подписи в `field_enums` (`label` / `value`)

Для полей с **фиксированным набором кодов** в CSV/БД в массиве **`options`** задаются объекты **`{ "label": "…", "value": "…" }`**: в выпадающем списке на карточке строки и в мастере показывается **`label`**, при сохранении в ячейку уходит **`value`** (как и для строковых элементов без `label`). Клиент **`row_editor.js`** и **`wizard_contest.js`** поддерживают оба формата. Если нужно не выпадающее меню, а **переключатель** для **двух** вариантов — см. ключ **`input_display`** в **разделе 4.4**. Фильтры списков (**`reward_type`**, **`contest_type`**) подтягивают подписи тем же способом через **`editor_config.flatten_field_enums`** и функции в **`sheet_list_display.py`**.

**Примеры правил в текущем `config.json` (не исчерпывающий список):**

| Лист | Колонка / контекст | Смысл подписей |
|------|---------------------|----------------|
| **GROUP**, **REWARD-LINK** | **`GROUP_CODE`** | `BANK` — «Страна»; `GOSB` — «ГОСБ»; `GROUPING` — «Группа»; `TB` — «Тер.банк». |
| **GROUP** | **`BASE_CALC_CODE`** | Пустое значение — «(пусто)»; далее те же четыре уровня, что и у **`GROUP_CODE`**. |
| **GROUP** | **`GET_CALC_METHOD`** | Значения **`1`**, **`2`**, **`3`** — человекочитаемые названия способов отбора (TOP-X, TOP-X%, по плану). |
| **INDICATOR** | **`CONTESTANT_SELECTION`** | **`0`** — «ВКО на дату операции»; **`1`** — «ВКО на конец турнира». |
| **REWARD** (`REWARD_ADD_DATA.newsType`) | **`newsType`** | **`AIPROMPT`** — «AI-новость»; **`TEMPLATE`** — «Новость по шаблону» (через `label`/`value`). |
| **CONTEST-DATA**, **TOURNAMENT-SCHEDULE**, **INDICATOR** | **`CALC_TYPE`** | **`1`** — «Ручной расчет»; **`0`** — «Промышленный расчет» (см. также **раздел 4.2** про значение **`3`** в выгрузке INDICATOR). |
| Разные листы | Флаги **`N`** / **`Y`** | В перечислениях используется единый порядок **«Да»** / **«Нет»** при сохранении **`Y`** / **`N`** в данные. |
| **CONTEST-DATA** и др. | **`CONTEST_TYPE`**, **`REWARD_TYPE`**, емкости, сегменты | Где нужно различать код и пояснение для оператора — через **`label`**. |
| **TOURNAMENT-SCHEDULE** | **`PERIOD_TYPE`** | Длинный список строковых значений (типы периодов турнира); в **`options`** порядок задан для удобства: **календарные месяцы** («турнир января» … «турнир декабря»), затем связанные с длительностью (**«март-июль»**, «турнир месяца», «турнир 2/3/4 месяца», «турнир 2/3 недели»), затем **кварталы**, **полугодия**, **год** («турнир года»), в конце — **«произвольный»**. |

Для путей внутри JSON-колонок (например **`REWARD_ADD_DATA`**, **`CONTEST_FEATURE`**) те же **`options`** могут быть привязаны к **`json_path`** в правиле **`field_enums`**; структура **`editor_field_ui`** с группой **`paths`** описана в таблице ключа **`editor_field_ui[]`** выше. Если для того же пути добавлено **`editor_field_numeric`**, выпадающий список **`field_enums`** для этого листа не используется.

**Динамические варианты из БД (`options_from_sheet`):** в том же правиле **`field_enums`** можно задать объект **`options_from_sheet`**: логический код листа **`source_sheet_code`** (из **`sheets`**), колонка **`value_column`**, подпись в списке — либо **`label_column`** (одна колонка), либо **`label_template`** — строка с плейсхолдерами **`{ИМЯ_КОЛОНКИ}`** как в Python **`str.format`** (литеральные **`{` / `}`** в тексте — **`{{`** / **`}}`**). Нужны **`value_column`** и **либо** **`label_column`**, **либо** **`label_template`** с хотя бы одним плейсхолдером. Необязательный список условий **`where`** (объект с **`equals`** — одно значение; с **`in`** — список), сортировка **`order_by`** (по умолчанию — по **`value_column`**). В выборку попадают только актуальные строки (**`is_current = 1`**). Серверный модуль **`field_enum_sheet_options.merge_field_enums_with_sheet_options`** подмешивает строки в **`options`** перед отдачей bootstrap и схемы мастера; блок **`options_from_sheet`** в JSON для клиента не передаётся. Статические элементы **`options`** из конфига добавляются после строк из БД, без дубликатов по **`value`**. Примеры: **`nonRewardCode`** — **`REWARD_CODE`** при **`REWARD_TYPE=ITEM`**, **`label_template`**: `"{FULL_NAME}: [\"{REWARD_CODE}\"]"`; **`rewardCode`** (в **`getCondition.rewards`**) — выборка при **`REWARD_TYPE=BADGE`** с тем же шаблоном подписи, плюс **`whitelist_validated_input`** (**раздел 4.5**). Валидация: **`config_validate.validate_field_enum_sheet_options`**.

### 4.4. Режим отображения перечисления: `input_display` и двухпозиционный переключатель

В каждом **правиле** внутри `field_enums` (в массиве `rules` сгруппированного блока или в плоском объекте с `sheet_code`) можно задать необязательное строковое поле **`input_display`**. Оно влияет только на **карточку строки** и **мастер конкурса** (клиент **`row_editor.js`**, мастер использует **`window.SpodYnField`** из того же файла). Сервер **`flatten_field_enums`** в `editor_config.py` передаёт правило на клиент **целиком**, включая **`input_display`**.

#### Назначение

- Явно выбрать **выпадающий список** (`<select>`) или **переключатель** (два состояния, подписи из `options`) для полей с **ровно двумя** вариантами в `options` и **`allow_custom`: false**.
- Не дублировать в коде жёсткую привязку «только Y/N»: порядок и подписи задаются в **`options`**, в ячейку CSV/БД по-прежнему пишется **`value`**.

#### Значения параметра `input_display`

| Значение | Поведение |
|----------|-----------|
| **`"toggle"`** | Если в правиле **ровно две** опции и **`allow_custom`** не включён — в UI показывается **переключатель** (не `<select>`). Первая опция в массиве **`options`** соответствует **левому** положению бегунка и **зелёному** фону трека; вторая — **правому** положению и **красному** фону. Подпись на треке — **`label`** текущего состояния. |
| **`"select"`** | Принудительно **выпадающий список**, даже если опций две. Удобно, если два варианта нужно показывать только как select. |
| **Ключ отсутствует** | **По умолчанию:** если опций **ровно две** и обе **`value`** после нормализации совпадают с классической парой **`N`** и **`Y`** (как у флагов «Нет»/«Да» в выгрузке) — показывается **переключатель**; если две опции с **другими** кодами (например **`all`** / **`one`**) — **список**, пока явно не задано **`"toggle"`**. Если опций **три и больше** — **всегда список**, даже при **`input_display`: "toggle"`** (клиент **`useToggleForEnumRule`** в **`row_editor.js`**). |

#### Ограничения и внешний вид переключателя (клиент)

- Ширина переключателя **не превышает 50% ширины блока** поля (контейнер ищется как ближайший предок с классами вроде **`.json-yn-band-item`**, **`.json-leaf-row`**, **`.scalar-cell__yn-band-item`** и т.п.); обёртка **`.spod-yn-wrap`** имеет **`width: fit-content`**, **`max-width: 50%`**, чтобы не растягиваться на всю сетку.
- Ширина **полосы под текст** задаётся CSS-переменной **`--spod-yn-text-px`**: значение должно быть **с единицей `px`** (иначе выражение **`width: calc(var(--spod-yn-text-px) + …)`** в браузере невалидно). После вставки в DOM измеряется **реальная ширина** подписей (`scrollWidth` у скрытого probe), а не оценка по числу символов (для кириллицы и жирного шрифта оценка «в ch» занижала ширину).
- **Фиксация размеров без «прыжков»:** пересчёт **не** выполняется отдельно на каждом экземпляре после `fonts.ready` и многократных `requestAnimationFrame`. Вместо этого один батч **`flushSpodYnToggleLayouts(root)`** обходит все **`.spod-yn-wrap`** внутри `root` после **`document.fonts.ready`** и двойного **`requestAnimationFrame`** вызывает **`applySpodYnToggleLayout`** и **`bindSpodYnToggleResizeGuard`**. Типичные точки вызова: конец **`activateDraftVariant`** (карточка строки — весь `document.body`), пересборка **`REWARD_ADD_DATA`** при смене **`REWARD_TYPE`**, конец **`render()`** мастера, возврат из режима «Сырой JSON» в «По полям», добавление строки в **`json_scalar_array`**, мультистраничный лист **GROUP** после монтирования JSON.
- **Изменение размера окна:** на контейнере поля один **`ResizeObserver`**; повторный расчёт в **`applySpodYnToggleLayout`** только если ширина блока изменилась **не меньше чем на 24 px** (порог в коде), плюс небольшая задержка (debounce), чтобы не реагировать на шум вёрстки.
- Если одна строка **не помещается** в половину блока, включается режим **`data-spod-yn-multiline="1"`**: подпись может переноситься до **трёх строк** по высоте (ограничение **`max-height`** в стилях), шрифт чуть меньше; бегунок **`.spod-yn-thumb`** масштабируется через **`clamp`** на треке.
- Публичный API для мастера и внешних страниц: **`window.SpodYnField`** — **`useToggleForEnumRule`**, **`isYnBinaryEnumRule`** (алиас), **`buildSpodYnToggleDom`**, **`syncSpodYnToggleVisual`**, **`flushSpodYnToggleLayouts`**, **`canonicalYnFromString`**, **`ynLabelsFromFieldEnumRule`**.

**1. Явный переключатель для пары «Да» / «Нет» (значения `Y` / `N`):**

### 4.5. Массивы объектов в JSON (`json_object_array`) и расширения перечислений

#### `editor_textareas`: `json_object_array`

В **`hints`** для JSON-колонки (например **`REWARD_ADD_DATA`**) задаётся:

| Ключ | Назначение |
|------|------------|
| **`json_path`** | Путь к **корню массива** объектов, например `["getCondition", "nonRewards"]`. |
| **`json_object_array`** | **`true`** — массив обрабатывается как список однотипных объектов, а не как вложенное дерево по индексам. |
| **`object_array_item_keys`** | Список строк — **имена полей в каждом элементе** (порядок = порядок полей в форме), например `["nonRewardCode"]` или `["rewardCode", "amount"]`. |
| **`array_allows_empty`** | Если **`false`**, нельзя удалить последнюю строку (остаётся минимум одна запись). |
| **`array_max_items`** | Верхняя граница числа элементов (кнопка «Добавить» блокируется при достижении). |

**Клиент (`row_editor.js`):** один блок-хост на корень массива; внутри — строки «одна запись = один объект» с кнопками **«Удалить»** и **«Добавить запись»**; после добавления/удаления пересчитываются пути **`data-json-path`** с числовым индексом. Сборка JSON в **`buildJsonFromFields`** сначала заполняет листья по путям, затем **перезаписывает** весь массив по данным хоста (**`data-json-object-array-host`**), чтобы в БД уходил корректный **`[]`**. Пустой массив в данных и объекты **без части ключей** допускаются: недостающие ключи нормализуются в пустые строки при развёртке.

**Примеры фрагментов `config.json`:**

```json
{
  "column": "REWARD_ADD_DATA",
  "json_path": ["getCondition", "nonRewards"],
  "json_object_array": true,
  "object_array_item_keys": ["nonRewardCode"],
  "array_allows_empty": true,
  "array_max_items": 32,
  "min_rows": 2,
  "max_rows": 12
}
```

```json
{
  "column": "REWARD_ADD_DATA",
  "json_path": ["getCondition", "rewards"],
  "json_object_array": true,
  "object_array_item_keys": ["rewardCode", "amount"],
  "array_allows_empty": true,
  "array_max_items": 32,
  "min_rows": 2,
  "max_rows": 12
}
```

```json
{
  "column": "REWARD_ADD_DATA",
  "json_path": ["itemGroupAmount"],
  "json_object_array": true,
  "object_array_item_keys": ["itemParam", "itemParamAmount"],
  "array_allows_empty": true,
  "array_max_items": 32,
  "min_rows": 2,
  "max_rows": 12
}
```

**`editor_field_ui`:** для подписей достаточно **одного** правила на поле **без индекса** в пути, например `["getCondition", "nonRewards", "nonRewardCode"]` — клиент **`findFieldUi`** сопоставляет пути вида `…, 0, nonRewardCode` со свёрткой индекса. Для массивов примитивов можно использовать wildcard-путь `["ключ", "*"]` и шаблоны `{index}` / `{index1}` в тексте, чтобы не хранить отдельные правила `["ключ", 0]`, `["ключ", 1]`, … .

#### `field_enums`: `whitelist_validated_input`

Если в правиле задано **`"whitelist_validated_input": true`** (и в **`options`** есть допустимые значения — в т.ч. после **`options_from_sheet`**), в режиме **«По полям»** для этого листа JSON показывается **однострочный `<input type="text">`**, а не `<select>`. Поведение подсветки (классы в **`app.css`**, обёртка **`.json-leaf-validated-wrap`**):

| Состояние ввода | Внешний вид |
|-----------------|-------------|
| Пусто (после **`trim`**) | Обычный нейтральный вид. |
| Непусто, значение **нет** в множестве **`value`** из **`options`** | Серый фон, **красный** текст (частичный ввод или опечатка). |
| Непусто, значение **совпало** с одним из **`value`** | Светло-зелёный фон, **зелёный** текст. |

Сервер **не** блокирует сохранение «красного» значения — только визуальная подсказка оператору.

**Примеры:**

1. **`getCondition.rewards.rewardCode`** — список кодов из БД (**`REWARD_TYPE=BADGE`**), **`whitelist_validated_input`**: true — ввод кода бейджа с подсветкой.

2. **`itemGroupAmount.itemParamAmount`** — статический список строк **`"0"` … `"100"`** + **`whitelist_validated_input`**: true — ввод лимита 0…100 как текста с той же подсветкой.

Для **`getCondition.rewards.amount`** используется обычный **`field_enums`** без whitelist — **`select`** с вариантами **`1`**, **`2`**, **`3`**.

#### Связка модулей

- **`src/field_enum_sheet_options.py`** — **`label_template_placeholders`**, **`format_label_from_template`**, выборка **`label_column`** или шаблона в **`_fetch_options_from_sheet`**.
- **`src/config_validate.py`** — для **`options_from_sheet`** требуется **`value_column`** и пара **`label_column` / `label_template`**.
- **`src/editor_config.py`** — в докстрингах упомянуты **`json_object_array`**, **`whitelist_validated_input`**.

## 6c. Списки листов: колонки, глобальные фильтры по связям и поиск

**С версии 0.2.42** все листы со списком (**CONTEST-DATA**, **GROUP**, **INDICATOR**, **REWARD**, **REWARD-LINK**, **TOURNAMENT-SCHEDULE**) используют один обработчик **`sheet_list`** и одну форму в **`sheet_list.html`**: классы **`sheet-list-form-reward sheet-list-form-global`** на **`<form>`** (без класса **`search`** — иначе поле поиска уезжает вправо). Фильтры — параметры **`gf_*`** (несколько одноимённых значений на измерение); логика пересечения множеств **`CONTEST_CODE`** и взаимного сужения вариантов — в **`src/global_sheet_filters.py`** (план: **`Docs/ПЛАН_ГЛОБАЛЬНЫЕ_ФИЛЬТРЫ_СПИСКОВ_ПО_СВЯЗЯМ.md`**). Ранее использовавшиеся отдельные query-параметры **`contest_type`**, **`reward_type`**, **`season`**, **`calc_mode`** **не** используются.

Блок фильтров в **`<details>`** (по умолчанию раскрыт): компактная сетка чипов (**`app.css`**: **`.global-filters-*`**, **`.sheet-list-filter-*--compact`**).

<a id="readme-6c1"></a>
### 6c.1. Порядок обработки запроса

1. **Загрузка строк из БД.** Из таблицы листа **`spod_sheet_<КОД>`** выбираются актуальные версии строк (`is_current = 1`, порядок `sort_key`, `row_index`, `id`). Для каждой строки читается полный набор ячеек через **`sheet_storage.row_to_cells`**.

2. **Справочники и индекс связей — один раз на запрос.** Функция **`sheet_list_display.build_lookup_tables(conn, CFG)`** строит именованные карты из **`config.json` → `sheet_list_lookups`**, затем индексы по TOURNAMENT-SCHEDULE и REWARD-LINK и возвращает словарь **`lu`**:
   - **`contest_full`** — `CONTEST_CODE` → `FULL_NAME` конкурса;
   - **`reward_full`** — `REWARD_CODE` → `FULL_NAME` (используется при подписях на других листах);
   - **`tournaments_for_contest`** — список турниров по конкурсу;
   - **`reward_links_by_contest`** — все записи REWARD-LINK по `CONTEST_CODE` (в т.ч. для списка INDICATOR);
   - **`reward_links_by_reward`** — записи REWARD-LINK, сгруппированные по **`REWARD_CODE`**: для каждой награды — список словарей с **`CONTEST_CODE`**, **`GROUP_CODE`**, **`REWARD_CODE`**.

3. **Глобальные фильтры (`gf_*`).** На каждый запрос строится **`GlobalFilterIndex`**: плоские поля и JSON-фрагменты **CONTEST-DATA**, уровни **GROUP**, показатели **INDICATOR**, поля и **TARGET_TYPE.seasonCode** у **TOURNAMENT-SCHEDULE**, тип и **REWARD_ADD_DATA** у наград, связанных с конкурсом через **REWARD-LINK**. Выбор читается **`selection_from_request`** (whitelist по universe из данных). Множество допустимых **`CONTEST_CODE`** — пересечение условий по всем измерениям с непустым выбором (**И** между параметрами, **ИЛИ** внутри одного параметра). Строка **REWARD** проходит, если награда связана хотя бы с одним конкурсом из этого множества. Варианты чипов для измерения **X** пересчитываются по конкурсам, уже удовлетворяющим **всем фильтрам кроме X** (взаимное сужение). Подписи для **`gf_contest_type`** и **`gf_reward_type`** подтягиваются из **`field_enums`** через **`contest_type_filter_options`** / **`reward_type_filter_options`**.

4. **Подписи и колонки для таблицы.** Для каждой строки, прошедшей фильтры листа, вызывается **`sheet_list_display.display_for_sheet_row(code, cells, lu)`**. Для **CONTEST-DATA** в объекте отображения есть **`contest_type_col`** — значение **`CONTEST_TYPE`** из ячеек (колонка «Тип конкурса»). Для **REWARD** функция возвращает в т.ч.:
   - **`primary_key`** — **`REWARD_CODE`** (колонка «Ключ» в таблице);
   - **`reward_name_col`** — **`FULL_NAME`** из **той же** строки награды (колонка «Награда»);
   - **`reward_contest_codes_col`** — уникальные непустые **`CONTEST_CODE`** из всех REWARD-LINK с данным **`REWARD_CODE`**, в порядке первого появления при обходе связей, через запятую;
   - **`title_line`** — названия конкурсов из **CONTEST-DATA** (`FULL_NAME` по каждому уникальному **`CONTEST_CODE`** из **REWARD-LINK** для данной награды, через « · »; колонка «Название / описание»);
   - **`group_codes_col`** — уникальные непустые **`GROUP_CODE`** из всех REWARD-LINK с данным **`REWARD_CODE`**, в порядке первого появления при обходе связей, через запятую;
   - **`relations_line`** — если есть связи: префикс **`Конкурс: `** и перечисление **уникальных** непустых **`CONTEST_CODE`** из тех же REWARD-LINK через запятую; иначе пустая строка.

   Для листа **REWARD-LINK** (одна строка таблицы = одна актуальная строка листа в БД) **`display_for_sheet_row`** дополнительно заполняет поля, из которых шаблон строит **четыре** видимые колонки (колонка «Связи» для этого листа в списке **не рендерится**):
   - **`reward_link_reward_code`** — значение **`REWARD_CODE`** из ячейки строки;
   - **`reward_link_group_code`** — **`GROUP_CODE`**;
   - **`reward_link_reward_name`** — **`FULL_NAME`** награды из справочника **`reward_full`** по **`REWARD_CODE`**;
   - **`reward_link_contest_name`** — **`FULL_NAME`** конкурса из **`contest_full`** по **`CONTEST_CODE`**;
   - **`reward_link_contest_code`** — код конкурса; в **таблицу** не выводится, но попадает в **`search_blob`** вместе с остальными ключами **`disp`** (и с полным JSON **`cells`**, где тоже есть **`CONTEST_CODE`**), поэтому поиск по подстроке **`q`** находит строки и по коду конкурса.

5. **Поиск по подстроке `q`.** Функция **`search_blob(cells, disp)`** собирает одну строку в **нижнем регистре**: сериализованный JSON всех ячеек строки плюс текстовые поля из результата **`display_for_sheet_row`**, включая **`title_line`**, **`reward_name_col`**, **`reward_contest_codes_col`**, **`group_codes_col`**, а для **REWARD-LINK** — в том числе **`reward_link_contest_code`**, **`reward_link_reward_code`**, **`reward_link_group_code`**, названия награды и конкурса. Строка попадает в итоговый список, только если после **`strip()`** параметр **`q`**, приведённый к нижнему регистру, **входит как подстрока** в **`search_blob`**. Для **REWARD** в подписи попадают и названия конкурсов из **`title_line`**, и полный JSON награды (в т.ч. **`REWARD_DESCRIPTION`** в ячейках).

6. **Схема колонок таблицы.** Для текущего листа **`app._sheet_list_columns_for_sheet(cfg, sheet_code)`** возвращает список словарей (**`key`**, **`label`**, **`cell_class`**, **`show`**, опционально **`value`**): из **`config.json` → `sheet_list_columns`** (блок с тем же **`sheet_code`**), иначе — запасной набор в коде. В шаблон передаётся **`app._visible_sheet_list_columns(...)`** — только колонки с **`show`** не равным **`false`**. Параллельно **`app._sheet_list_consistency_column_for_sheet`** формирует **`consistency_column`** (**`show`**, **`label`**, **`cell_class`**). Списки передаются в шаблон (см. **раздел 6c.6**).

7. **Ответ шаблону.** В **`sheet_list.html`** передаются **`sheet_code`**, **`sheet_title`**, **`rows`**, **`list_columns`**, **`consistency_column`** (**`show`**, **`label`**, **`cell_class`** из **`config.json`** или значения по умолчанию), **`q`**, **`global_filter_blocks`** (список блоков с **`label`**, **`param`**, **`options`**, **`selected_list`**) и флаг **`apply_global_filters`**.

### 6c.2. Интерфейс: фильтр и поиск

- Под заголовком листа — краткая подсказка по колонкам.
- **Форма** с классами **`sheet-list-form-reward sheet-list-form-global`**: сверху **`<details class="global-filters-details">`** с сеткой **`global-filters-grid`** из компактных блоков чипов, под ним строка **`sheet-list-search-full`** (поле **`q`** + «Найти»).
- **Мультивыбор:** для каждого измерения — **`sheet-list-filter-row`** с иконками «все / снять» и **`sheet-list-filter-chips-wrap--compact`**; **`name`** = соответствующий **`gf_*`**. При изменении чекбокса — **`form.submit()`** (**GET**). Кнопка **«Сбросить все фильтры»** снимает отметки со **всех** чекбоксов **`gf_*`** и снова отправляет форму (текст в поле **`q`** сохраняется).
- **Кнопки «все / снять»** выделяют или снимают все чекбоксы в соответствующем **`data-sheet-list-filter-target`**, затем отправляют форму. «Снять все» даёт пустой набор параметров в query — без фильтра.
- Стили: **`src/static/app.css`** — блок списков с фильтрами-чипами (`.sheet-list-form-reward`, `.sheet-list-filter-row`, `.sheet-list-filter-chip`, `.sheet-list-search-full` и т.д.).
- Таблица списка **REWARD-LINK**: на элемент **`<table>`** добавляется класс **`.data-list-reward-link`** (рядом с **`.data-list-wide`**). Задано **`table-layout: fixed`**, **`min-width: 0`** у ячеек; колонки **«Код награды»** и **«Уровень»** — ограниченная ширина в **`ch`**, **`text-overflow: ellipsis`** для длинных кодов; **«Название награды»** и **«Название турнира»** — доли ширины (~**36%** каждая) с **`min-width`**, перенос длинного текста (**`overflow-wrap`**, **`word-break`**). Для **`.data-list-reward-link .cell-mono`** снят избыточный потолок **`max-width`**, заданный у общего **`.data-list-wide .cell-mono`**, чтобы узкие колонки не разъезжались.

### 6c.3. Связь с доменной моделью

Связь награды с конкурсом и группами задаётся листом **REWARD-LINK** (см. **`database_model.logical_relationships_ru`** и **раздел 4.1** README). В списке REWARD для одного **`REWARD_CODE`** агрегируются все связанные **`CONTEST_CODE`** и **`GROUP_CODE`**, чтобы в одной строке таблицы было видно, к каким конкурсам и группам привязана награда по данным выгрузки.

<a id="readme-6c4"></a>
### 6c.4. Карточка строки REWARD и список

Список листа **не заменяет** блок «Связи» на странице редактирования строки: там по-прежнему используется **`relations.build_context_for_row`** (в т.ч. блок **REWARD-LINK** со ссылками на конкретные строки связей). Ссылка **«Просмотр»** ведёт на **`/sheet/REWARD/row/{id}?from_list=1`** для корректной цепочки возврата в **`trail_nav.js`**.

<a id="readme-6c5"></a>
### 6c.5. Список REWARD-LINK и карточка строки связи

В списке **REWARD-LINK** кнопка **«Просмотр»** открывает **`/sheet/REWARD-LINK/row/{id}?from_list=1`** — редактирование **конкретной** строки связи (пара конкурс — группа — награда). На карточке по-прежнему доступны все поля CSV этой строки, включая **`CONTEST_CODE`**, даже если в таблице списка колонка с кодом конкурса скрыта. Цепочка возврата в **`trail_nav.js`** учитывает **`from_list=1`** так же, как для других листов.

<a id="readme-6c6"></a>
### 6c.6. Конфигурация колонок таблицы списка (`sheet_list_columns`, `sheet_list_lookups`)

В **`config.json`** задаются:

1. **`sheet_list_lookups`** — именованные справочники для подстановки **значения с другого листа** по ключу из текущей строки (например **`contest_full`**: `CONTEST_CODE` → `FULL_NAME` с листа **CONTEST-DATA**). Элементы: **`id`**, **`source_sheet`**, **`key_column`**, **`value_column`**, опционально **`comment_ru`**.

2. **`sheet_list_columns`** — для каждого листа со списком: **`sheet_code`**, опционально **`consistency_column`**, **`rules`**.

**`consistency_column`** (в том же блоке, что и **`sheet_code`**): **`show`** — показывать ли колонку с бейджами OK / «Проблемы» по **`r.ok`**; **`label`** — заголовок **`<th>`**; **`cell_class`** — классы **`<th>`** и **`<td>`** (по умолчанию **`col-flag`**). Для **CONTEST-DATA** и **GROUP** в репозитории задано **`show`: false**; для остальных листов со списком — **`true`**. Значение берётся из **`app._sheet_list_consistency_column_for_sheet`**. Опционально **`comment_ru`**.

Каждый элемент **`rules[]`** — объект:

| Поле | Назначение |
|------|------------|
| **`key`** | Имя атрибута в объекте строки **`r`**, под которым в шаблоне читается значение (**`r.get(key)`**). |
| **`label`** | Текст заголовка **`<th>`**. |
| **`cell_class`** | CSS-классы ячейки — см. **`src/static/app.css`**. |
| **`show`** | Если **`false`**, колонка **не выводится** в таблице списка (правило можно оставить в конфиге). По умолчанию **`true`**. Значение ячейки по-прежнему вычисляется (**`apply_configured_list_column_values`** обрабатывает все правила листа). |
| **`value`** | После сборки строки списка **`app.sheet_list`** вызывает **`sheet_list_column_values.apply_configured_list_column_values`**: по этому объекту **перезаписывается** **`r[key]`**. Источник задаётся полем **`kind`**: |

**Варианты `value.kind`:**

| `kind` | Назначение |
|--------|------------|
| **`cell`** | Значение из ячейки текущей строки: **`column`** — имя колонки CSV. |
| **`lookup`** | **`lookup_id`** — имя из **`sheet_list_lookups`** (карта в **`lu[lookup_id]`**); **`key_from_column`** — колонка текущей строки, по которой ищется ключ (например подпись конкурса по **`CONTEST_CODE`**). |
| **`json_leaf`** | Лист JSON в колонке **`column`**, путь **`path`** (массив ключей); для **`TARGET_TYPE` / `seasonCode`** используется тот же смысл, что и **`target_type_season_code`**. |
| **`display_field`** | Поле из результата **`display_for_sheet_row`** — **`field`**: имя ключа (например **`relations_line`**, **`title_line`**, **`reward_name_col`**). Сложная логика между листами остаётся в **`sheet_list_display.display_for_sheet_row`**; в конфиге явно задаётся, **какое поле** показать в колонке. |
| **`builtin`** | Зарегистрированные сценарии в **`sheet_list_column_values`**: для агрегированного списка **GROUP** — **`group_list_contest_code`**, **`group_list_contest_name`**, **`group_list_relations`**. |

**После `rules`:** если **`consistency_column.show`**, шаблон **`sheet_list.html`** выводит колонку консистентности (подпись и классы из конфига); затем фиксированная колонка **«Просмотр»**.

**Особый случай шаблона:** для **INDICATOR** при **`title_line`** и наличии **`subtitle_line`** выводятся две строки (см. **`sheet_list.html`**) — **`subtitle_line`** по-прежнему приходит из **`display_for_sheet_row`**.

**Поиск по `q`:** **`search_blob`** по-прежнему строится из ячеек и **`display_for_sheet_row`** (шаг 5 в **разделе 6c.1**), не из произвольных `value` в конфиге.

При отсутствии блока для листа или пустом **`rules`** используется запасной набор в **`_sheet_list_columns_for_sheet`** в **`app.py`**. Валидация: **`config_validate.validate_sheet_list_lookups`**, **`validate_sheet_list_column_values`** (предупреждения в лог при старте).

<a id="readme-6c7"></a>
### 6c.7. Пошаговое руководство: добавить колонку в таблицу списка

Ниже — порядок действий и все типичные варианты источника данных (текущий лист, другой лист, JSON, цепочка через **REWARD-LINK**). Цель — добавить элемент в **`sheet_list_columns[].rules`** для нужного **`sheet_code`** и при необходимости расширить конфиг или код.

#### Общий порядок (чек-лист)

1. **Определить источник текста** для ячейки: плоская колонка текущего CSV, лист JSON, справочник «ключ → подпись» с другого листа, уже вычисленное поле из **`display_for_sheet_row`**, или отдельная логика (**`builtin`** / правка Python).
2. В **`config.json`** в блоке **`sheet_list_columns`** найти объект с **`sheet_code`** целевого листа (или создать объект целиком: **`sheet_code`**, **`consistency_column`**, **`rules`**).
3. В массив **`rules`** **вставить новый объект** в нужной позиции: порядок в массиве = порядок колонок в таблице (сразу после колонки **#**).
4. Заполнить поля правила:
   - **`key`** — уникальный идентификатор колонки в объекте строки **`r`**; по нему шаблон **`sheet_list.html`** выводит **`r.get(key)`** (см. исключение для **`title_line`** + **`subtitle_line`** ниже).
   - **`label`** — заголовок **`<th>`**.
   - **`cell_class`** — CSS-классы ячейки и заголовка (см. подраздел про стили).
   - **`value`** — объект с **`kind`** и полями по схеме (см. ниже).
5. Перезапустить приложение (конфиг читается при старте), проверить лог на предупреждения валидации (**`validate_sheet_list_lookups`**, **`validate_sheet_list_column_values`**).
6. Если нужен **поиск по `q`** по новому тексту — убедиться, что текст попадает в **`search_blob`** (см. подраздел «Поиск»).

#### Поля **`key`** и **`cell_class`**: что обязательно задать, куда **не** нужно ничего прописывать

| Вопрос | **`key`** | **`cell_class`** |
|--------|-----------|------------------|
| Где задаётся? | Только в **`config.json`** → **`sheet_list_columns[].rules[]`** у **каждого** правила колонки. | Только в **`config.json`** в том же правиле — одна строка, несколько классов **через пробел** (например **`"cell-wrap cell-muted"`**). |
| Нужен ли «реестр» или список допустимых значений в коде? | **Нет.** Отдельного файла со списком **`key`** в проекте нет. | **Нет.** Имена классов **нигде заранее не перечисляются** — шаблон подставляет строку в атрибут **`class`** как есть. |
| Нужно ли править **`sheet_list.html`**? | **Обычно нет.** Исключение: если нужна **такая же** особая вёрстка, как у **`title_line`** + **`subtitle_line`** (две строки в одной ячейке) — тогда либо используйте **`key`**: **`title_line`** на листе **INDICATOR**, либо дорабатывайте шаблон под **свой** **`key`** (редкий случай). | **Нет.** Классы уже выводятся из конфига: **`{{ c.cell_class }}`** на **`<th>`** и **`<td>`**. |
| Нужно ли править **`app.css`**? | Не из‑за самого **`key`**. | **Только если** вы вводите **новое** имя класса и хотите отличать колонку по ширине/цвету/переносу. Если используете **только классы из таблицы ниже** — правки CSS **не обязательны**. |

**Про **`key`** подробнее:**

- Значение **`key`** — это имя поля в объекте строки **`r`**, куда после расчёта попадает текст ячейки; шаблон читает **`r.get(c.key, "")`**.
- **`key`** должен быть **уникален внутри массива `rules`** данного листа: два правила с одним **`key`** перезапишут одно и то же поле **`r`**.
- «Новые» **`key`** **автоматически ниоткуда не подхватываются** — их **вы придумываете** в JSON (латиница/подчёркивания, например **`schedule_period_col`**). Главное — чтобы **`value`** (или логика в Python) заполнили **`r[key]`**.

**Про **`cell_class`** подробнее:**

- Это **не** идентификатор колонки в данных, а **только оформление**: та же строка попадает в HTML как список CSS-классов.
- Если задать выдуманное имя (**`my_col_style`**) без правил в **`app.css`**, колонка **будет без специального вида** (наследуются общие стили таблицы **`.data`** / **`.data-list-wide`**).
- Модификаторы таблицы (**`data-list-contests`**, **`data-list-contests--contest-data`**, **`data-list-reward-link`**) задаются в **`sheet_list.html`** по **`sheet_code`** — от **`cell_class`** не зависят; узкие/широкие колонки для **CONTEST-DATA** / **REWARD-LINK** дополнительно подкручиваются классами вида **`col-contest-code`**, **`col-reward-link-code`** (см. таблицу ниже).

#### Полный справочник классов, используемых в `cell_class`

Все классы ниже уже используются в **`config.json` → `sheet_list_columns[].rules[].cell_class`**. Их можно комбинировать через пробел.  
Колонки **#**, **«Консистентность»**, **«Просмотр»** оформляются отдельно (в шаблоне/`consistency_column`) и не входят в `rules`.

| Класс | Где применяется сейчас | Что делает (текст / размер / поведение) |
|------|-------------------------|-------------------------------------------|
| **`cell-code`** | Коды **`CONTEST_CODE`**, **`REWARD_CODE`**, **`TOURNAMENT_CODE`** в списках | Моношрифт, **очень жирный** (`font-weight: 900`), без переноса (`nowrap`), выравнивание текста влево, вертикально по центру ячейки (`td.cell-code`), рассчитан под «кодовые» колонки. |
| **`cell-code-noshrink`** | Коды наград (**`REWARD_CODE`**) в списках **REWARD** и **REWARD-LINK** | «Несжимаемый» код: запрещает перенос и разрывы, не уводит код в `...`; строка кода отображается целиком (при нехватке места список уходит в горизонтальный скролл контейнера). |
| **`cell-mono`** | Коды второго уровня (например `GROUP_CODE`, `seasonCode`, `INDICATOR_CODE`) | Моношрифт, компактный размер, ограниченный `max-width`; для коротких кодов и технических значений. |
| **`cell-wrap`** | Основные текстовые колонки (`title`, `name`, и т.д.) | Базовый текстовый стиль с переносом длинных слов, нормальная читаемость многострочного текста. |
| **`cell-muted`** | Второстепенный текст, чаще «Связи» | Приглушённый цвет и меньший кегль для менее важного контента. |
| **`cell-relations-pre`** | Колонки «Связи» с форматированием в несколько строк | `white-space: pre-line` — сохраняет переносы строк из значения. |
| **`cell-shrink-wrap`** | Текстовые колонки, которые можно сжимать | Разрешает сжимать колонку и переносить текст на новые строки (`overflow-wrap: anywhere`, `word-break: break-word`). |
| **`cell-shrink-ellipsis`** | Колонки, которые можно сжимать без переноса | При уменьшении ширины не переносит текст, а аккуратно обрезает с `...` (`text-overflow: ellipsis`). |
| **`cell-h-left`** | Все колонки списков (через `cell_class`) | Явное выравнивание текста **по левому краю** (`text-align: left`). |
| **`cell-h-center`** | Доступен для точечного применения | Выравнивание текста **по центру** (`text-align: center`). |
| **`cell-h-right`** | Доступен для точечного применения | Выравнивание текста **по правому краю** (`text-align: right`). |
| **`cell-w-strong`** | Преимущественно кодовые колонки | Усиленная насыщенность текста (`font-weight: 900`) для акцентных значений. |
| **`cell-w-medium`** | Основные текстовые колонки | Средняя насыщенность текста (`font-weight: 500`). |
| **`cell-w-thin`** | Второстепенные/служебные колонки | Тонкая насыщенность текста (`font-weight: 400`). |
| **`col-contest-code`** | CONTEST-DATA/GROUP | Настраивает ширину кодовой колонки конкурса (фиксированный диапазон), без наложения на соседние колонки. |
| **`col-contest-name`** | CONTEST-DATA/GROUP | Ширина/доля колонки названия конкурса. |
| **`col-contest-type`** | CONTEST-DATA | Размер колонки типа конкурса; обычно сочетается с `cell-shrink-ellipsis`. |
| **`col-contest-relations`** | GROUP | Геометрия колонки «Связи» для списка GROUP. |
| **`col-reward-link-code`** | REWARD-LINK (`REWARD_CODE`) | Фиксированная кодовая ширина в таблице REWARD-LINK. |
| **`col-reward-link-level`** | REWARD-LINK (`GROUP_CODE`) | Узкая колонка уровня, с обрезкой при нехватке места. |
| **`col-reward-link-contest-code`** | REWARD-LINK (`CONTEST_CODE`) | Отдельная геометрия кода конкурса в списке связей, чтобы код не налезал на соседние колонки. |
| **`col-reward-link-name`** | REWARD-LINK (названия) | Базовый стиль «широких» колонок названий в REWARD-LINK. |
| **`col-reward-link-name--reward`** | REWARD-LINK (название награды) | Доля ширины под название награды. |
| **`col-reward-link-name--contest`** | REWARD-LINK (название конкурса) | Доля ширины под название конкурса. |

#### Разделители между колонками (тонкие пунктирные линии)

Для таблиц списков включён еле заметный вертикальный разделитель между колонками:

- `table.data th, table.data td` → `border-right: 1px dotted ...`;
- у последней колонки `border-right` отключён.

Это визуально отделяет соседние колонки, не делая сетку тяжёлой.

#### Глобальные правила выравнивания и раскладки списков

- Для всех таблиц списка (**`.data-list-wide`**) включено вертикальное центрирование ячеек: `th/td { vertical-align: middle; }`.
- Горизонтальное выравнивание регулируется классами `cell-h-left` / `cell-h-center` / `cell-h-right`; если класс не задан — применяется базовое выравнивание таблицы.
- Для списка **REWARD-LINK** используется авто-раскладка без наложений (`table-layout: auto`, `width: max-content`, `min-width: 100%`): при нехватке ширины контент не перекрывает соседние колонки, а включается горизонтальный скролл контейнера.

#### Какие новые классы применены по листам (текущее состояние)

- **CONTEST-DATA / GROUP**: коды конкурса — `cell-code ... cell-h-left cell-w-strong`; текстовые колонки — `cell-wrap` + `cell-shrink-wrap`/`cell-shrink-ellipsis` + `cell-w-medium`; связи — `cell-w-thin`.
- **INDICATOR**: `contest_code` — `cell-code ... cell-w-strong`; `title_line`/`add_calc_type` — `cell-wrap` с управляемым сжатием; `relations_line` — тонкий вес `cell-w-thin`.
- **REWARD**: `REWARD_CODE` — `cell-code cell-code-noshrink ... cell-w-strong` (код не сокращается); текстовые поля — `cell-w-medium`; связи — `cell-w-thin`.
- **REWARD-LINK**: `REWARD_CODE` — `cell-code cell-code-noshrink col-reward-link-code`; `CONTEST_CODE` — `cell-code cell-shrink-ellipsis col-reward-link-contest-code`; названия — `cell-wrap cell-shrink-wrap`; уровень — `cell-mono`.
- **TOURNAMENT-SCHEDULE**: `TOURNAMENT_CODE`/`CONTEST_CODE` — `cell-code ... cell-w-strong`; статус/период — `cell-shrink-ellipsis`; название конкурса — `cell-shrink-wrap`.

#### Как выбирать класс для новой колонки

1. **Код/идентификатор**: `cell-code` (главные коды) или `cell-mono` (вторичные коды).
2. **Длинный текст с переносом**: `cell-wrap cell-shrink-wrap`.
3. **Компактный статус/короткая подпись с `...`**: `cell-wrap cell-shrink-ellipsis`.
4. **Связи в несколько строк**: `cell-wrap cell-muted cell-relations-pre cell-shrink-wrap`.
5. Если нужна геометрия конкретного листа (ширина/доля) — добавьте листовой `col-*` класс.

Для нового поведения создайте новый CSS-класс в **`src/static/app.css`**, затем используйте его в `cell_class`.

#### Вариант A — значение из плоской колонки **текущего** листа (`kind`: **`cell`**)

Подходит, когда в CSV/таблице SQLite у этого листа уже есть колонка с нужным значением.

1. В **`rules`** добавить, например:  
   `"value": { "kind": "cell", "column": "ИМЯ_КОЛОНКИ" }`  
   где **`ИМЯ_КОЛОНКИ`** — заголовок из импорта, совпадающий с **`sheets[].`** настройками и **`headers`** в БД.
2. **`key`** — любой новый уникальный ключ (например **`my_extra_col`**); **`label`** и **`cell_class`** — по смыслу.
3. Код менять **не** требуется. Поиск: значение обычно уже входит в **`search_blob`**, т.к. в него сериализуются **все** ячейки **`cells`** (JSON строки).

#### Вариант B — лист JSON в колонке текущей строки (`kind`: **`json_leaf`**)

Подходит для полей внутри **`json_columns`** (например **`TARGET_TYPE.seasonCode`** у **TOURNAMENT-SCHEDULE**).

1. Указать **`column`** — имя JSON-колонки в строке, и **`path`** — массив ключей/индексов до листа (например **`["seasonCode"]`**).
2. Для пары **`TARGET_TYPE`** + **`["seasonCode"]`** в коде используется та же семантика, что и **`target_type_season_code`** (**`sheet_list_display`** / **`sheet_list_column_values`**).
3. Поиск: если значение **только** в JSON, оно уже может находиться через **`json.dumps(cells)`** в **`search_blob`**; отдельно добавлять ничего не нужно.

#### Вариант C — подпись с **другого** листа по ключу из текущей строки (`kind`: **`lookup`**)

Типичный случай: в строке есть **`CONTEST_CODE`**, а показать нужно **`FULL_NAME`** конкурса с **CONTEST-DATA**, или **`REWARD_CODE`** → название награды с **REWARD**.

1. В **`sheet_list_lookups`** завести (или использовать существующий) справочник:
   - **`id`** — произвольное имя (например **`contest_full`**, **`reward_full`**);
   - **`source_sheet`** — код листа-источника (**`CONTEST-DATA`**, **`REWARD`**, …);
   - **`key_column`**, **`value_column`** — по какому полю искать и что подставлять в ячейку.
2. В правиле колонки:  
   `"value": { "kind": "lookup", "lookup_id": "<id>", "key_from_column": "КОЛОНКА_В_ТЕКУЩЕЙ_СТРОКЕ" }`  
   Например, для строки **REWARD-LINK**: **`key_from_column`**: **`CONTEST_CODE`**, **`lookup_id`**: **`contest_full`** — это и есть «подтянуть название конкурса по коду из связи».
3. **`validate_sheet_list_column_values`** проверяет, что **`lookup_id`** объявлен в **`sheet_list_lookups`**, и что для **`lookup`** задан **`key_from_column`**.
4. Поиск: подставленное значение может **не** дублироваться в **`search_blob`** отдельно, но код конкурса/награды в **`cells`** и поля **`disp`** для **REWARD-LINK** уже участвуют в поиске; при необходимости расширить **`search_blob`** (см. ниже).

**Связь через REWARD-LINK без отдельной колонки в «целевом» листе:**  
На листе **REWARD** в одной строке **нет** одного **`CONTEST_CODE`** — связи идут через множество строк **REWARD-LINK**. Поэтому «названия конкурсов через связи» уже собираются в **`display_for_sheet_row`** в поле **`title_line`** (и частично в **`relations_line`**). Для новой **агрегированной** колонки (свой формат текста по связям) нужно либо использовать существующие **`display_field`**, либо расширить **`display_for_sheet_row`** (вариант E).

#### Вариант D — поле из **`display_for_sheet_row`** (`kind`: **`display_field`**)

Используется, когда сложная логика уже реализована в **`src/sheet_list_display.py`** и возвращается в словаре **`disp`**.

1. В **`value`** задать: **`{ "kind": "display_field", "field": "<имя_ключа>" }`** — имя должно **точно** совпадать с ключом, который возвращает **`display_for_sheet_row(sheet_code, cells, lu)`** для этого листа.
2. Опорная таблица ключей **`disp`** по листам (для подбора **`field`** без чтения кода):

| Лист | Примеры ключей **`disp`** (неполный перечень) |
|------|-----------------------------------------------|
| **CONTEST-DATA** | **`primary_key`**, **`title_line`**, **`relations_line`**, **`contest_type_col`** |
| **GROUP** (в т.ч. до агрегата в списке) | **`primary_key`**, **`title_line`**, **`relations_line`** — для **агрегированного** списка GROUP см. вариант F |
| **INDICATOR** | **`title_line`**, **`subtitle_line`**, **`relations_line`**, **`contest_code`**, **`add_calc_type`**, **`indicator_code_col`**, … |
| **REWARD** | **`reward_name_col`**, **`reward_contest_codes_col`**, **`title_line`** (названия конкурсов через **REWARD-LINK**), **`group_codes_col`**, **`relations_line`**, … |
| **REWARD-LINK** | **`reward_link_reward_code`**, **`reward_link_group_code`**, **`reward_link_reward_name`**, **`reward_link_contest_code`**, **`reward_link_contest_name`**, … |
| **TOURNAMENT-SCHEDULE** | **`schedule_tournament_status_col`**, **`schedule_contest_code_col`**, **`schedule_period_col`**, **`schedule_contest_name_col`**, **`schedule_season_col`**, **`relations_line`**, … |

3. Если нужного ключа **нет** — добавить вычисление в **`display_for_sheet_row`** и при необходимости протащить значение в **`app.sheet_list`** в объект **`row_out`** (как для **INDICATOR** / **REWARD** / **REWARD-LINK** / **TOURNAMENT-SCHEDULE** — см. **`app.py`**, ветка **`sheet_list`**), затем указать **`display_field`** в конфиге.

#### Вариант E — новая **вычисляемая** колонка (правки Python)

Если ни **`cell`**, ни **`lookup`**, ни **`json_leaf`**, ни существующий **`display_field`** не выражают нужное:

1. Добавить поле в возвращаемый словарь в **`display_for_sheet_row`** в **`src/sheet_list_display.py`** (например **`my_custom_col`**). При необходимости использовать **`lu`**: **`contest_full`**, **`reward_full`**, **`reward_links_by_reward`**, **`reward_links_by_contest`**, **`tournaments_for_contest`** и т.д. (см. **`build_lookup_tables`**).
2. В **`app.py`** в **`sheet_list`** для соответствующего **`code`** добавить строку вида **`row_out["my_custom_col"] = disp.get("my_custom_col", "")`** по аналогии с **REWARD** / **REWARD-LINK**.
3. В **`sheet_list_columns.rules`** задать **`value`**: **`{ "kind": "display_field", "field": "my_custom_col" }`**.
4. **Поиск (`q`):** функция **`search_blob`** явно перечисляет поля из **`disp`**, которые участвуют в поиске. Если новое поле должно быть **искомым**, добавить его имя в кортеж в **`search_blob`** в **`sheet_list_display.py`**.

#### Вариант F — лист **GROUP** (агрегат «одна строка на конкурс»)

Для списка **GROUP** строки собираются особым образом: **`apply_configured_list_column_values`** вызывается с **`cells=None`**, **`disp=None`**, зато передаётся **`agg`** с **`contest_code`** и **`members`**. Работают только правила с **`kind`: `builtin`**: **`group_list_contest_code`**, **`group_list_contest_name`**, **`group_list_relations`** (**`sheet_list_column_values.py`**).

- Чтобы добавить **ещё одну колонку** для этого режима, в общем случае нужно: реализовать функцию в **`_BUILTIN`** в **`sheet_list_column_values.py`**, зарегистрировать имя в **`_BUILTIN`**, добавить имя в **`builtins_ok`** в **`config_validate.py`**, затем в **`config.json`** указать **`value`**: **`{ "kind": "builtin", "name": "..." }`**.

#### Особенности шаблона **`sheet_list.html`**

- Для колонки с **`key`** **`title_line`**, если у строки есть **`subtitle_line`** (лист **INDICATOR**), выводятся две строки в одной ячейке (основной текст + приглушённая подстрока). Для **других** **`key`** — только **`r.get(c.key)`**.
- Колонка **«Просмотр»** и колонка **консистентности** задаются **не** через **`rules`**, а через шаблон и **`consistency_column`**.

#### Стили: **`cell_class`**, модификаторы таблицы и **`STATIC_ASSET_VERSION`**

Перечень универсальных и листо-специфичных классов, правило «только **`config.json`** / при новом имени — **`app.css`**» — см. **подразделы выше** («**`key` и `cell_class`**», «**Универсальные классы**», «**Классы под конкретные листы**»).

Дополнительно:

- Модификаторы **`<table>`** (**`data-list-contests`**, **`data-list-contests--contest-data`**, **`data-list-reward-link`**) задаются в **`sheet_list.html`** по **`sheet_code`** — см. **раздел 6c.2**; от выбора модификатора зависят селекторы в **`app.css`** для **`col-contest-*`**, **`col-reward-link-*`**.
- Колонка **«Консистентность»** использует классы из **`consistency_column.cell_class`** в конфиге (часто **`col-flag`**), а не из **`rules`**.
- После **любого** изменения **`app.css`** имеет смысл поднять **`STATIC_ASSET_VERSION`** в **`app.py`**, чтобы браузер подтянул стили.

#### Поиск по строке **`q`**

- **`search_blob`** склеивает **полный JSON `cells`** и **фиксированный набор** полей из **`disp`** (см. **`search_blob`** в **`sheet_list_display.py`**).
- Значения, попавшие в ячейку таблицы **только** через **`apply_configured_list_column_values`** (перезапись **`r[key]`**), **не** добавляются в **`search_blob`** автоматически. Если текст из новой колонки должен искаться — либо он уже есть в **`cells`/JSON**, либо добавьте поле в **`disp`** и в список полей в **`search_blob`**, либо расширьте **`search_blob`** иначе.

## 6b. Мастер «Создать конкурс» (подробно)

<a id="readme-6b1"></a>
### 6b.1. Назначение и порядок шагов

Мастер ведёт пользователя по цепочке без записи в таблицы листов до последнего шага. После подтверждения на экране предпросмотра сервер выполняет `wizard_contest.commit_wizard`: вставка строк в `spod_sheet_*` (CONTEST-DATA, GROUP, …), затем `consistency.run_all_checks`; в режиме `strict` при ошибке на одной из вставленных строк выполняется откат транзакции.

| Шаг (индекс) | Содержание |
|----------------|------------|
| 0 | Плоские поля + JSON колонки листа **CONTEST-DATA** |
| 1 | Число строк **GROUP**, формы по каждой строке; `CONTEST_CODE` подставляется из шага 0 |
| 2 | **REWARD-LINK**: не меньше числа различных пар **`(GROUP_CODE, GROUP_VALUE)`** в шаге GROUP; связь с `CONTEST_CODE` |
| 3 | **REWARD**: по одному блоку на каждый уникальный `REWARD_CODE` из связей |
| 4 | **INDICATOR**: задаётся число строк, общий `CONTEST_CODE` |
| 5 | **TOURNAMENT-SCHEDULE**: число строк, общий `CONTEST_CODE` |
| 6 | Предпросмотр JSON и кнопка «Подтвердить создание в базе» |

Валидация обязательных полей и запрета пустых значений на шагах 0–5 выполняется в браузере по правилам `editor_field_ui` (`required`, `allows_empty`); у подписей полей отображаются текстовые бейджи **«Обязательно»** и **«Не пусто»** с пояснением в `title`.

<a id="readme-6b2"></a>
### 6b.2. Таблица `wizard_draft`

Создаётся при старте приложения (`ensure_wizard_draft_table`). Назначение — **промежуточное** состояние мастера, чтобы после сбоя или закрытия вкладки можно было продолжить.

| Колонка | Назначение |
|---------|------------|
| `id` | Суррогатный ключ |
| `draft_uuid` | Уникальный идентификатор сессии (UUID с клиента) |
| `step_index` | Текущий шаг (0…6) |
| `status` | Для черновиков мастера используется значение **`EDIT`** |
| `state_json` | JSON-объект полного состояния мастера (см. ниже) |
| `contest_code_preview` | Копия `CONTEST_CODE` из состояния для отображения в списке (может быть пусто) |
| `updated_at` | Метка времени UTC (`YYYY-MM-DDTHH:MM:SSZ`) |

Черновик **не** дублирует бизнес-данные в таблицах листов до `commit`.

<a id="readme-6b3"></a>
### 6b.3. Формат `state` для `PUT /wizard/new-contest/draft`

Тело запроса:

```json
{
  "draft_uuid": "строка-uuid",
  "step_index": 0,
  "state": { ... }
}
```

Объект **`state`** (внутри `state_json` на сервере) содержит поля:

| Поле | Тип смысла |
|------|------------|
| `stepIndex` | Номер шага (дублирует `step_index` запроса; при расхождении приоритет у поля запроса при записи `step_index` в таблицу) |
| `contest` | `{ "cells": { "COL": "значение", ... } }` |
| `groups`, `reward_links`, `rewards`, `indicators`, `schedules` | Массивы `{ "cells": { ... } }` |
| `groupCount`, `linkCount`, `indicatorCount`, `scheduleCount` | Числа для UI «сформировать N строк» |

<a id="readme-6b4"></a>
### 6b.4. API черновика (кратко)

- **`GET /wizard/new-contest/drafts`** — массив объектов `{ draft_uuid, step_index, contest_code_preview, updated_at }`.
- **`GET /wizard/new-contest/draft/{uuid}`** — 404, если черновик не найден.
- **`PUT /wizard/new-contest/draft`** — upsert по `draft_uuid` (SQLite `ON CONFLICT DO UPDATE`).
- **`DELETE /wizard/new-contest/draft/{uuid}`** — удаление (кнопка «Удалить» у строки в списке на странице мастера).

<a id="readme-6b5"></a>
### 6b.5. Поведение клиента (`wizard_contest.js`)

- При открытии `/wizard/new-contest` без параметров: запрашивается список черновиков; при наличии — панель с кнопками «Продолжить» (переход на `?draft=uuid`), «Удалить», «Начать новый конкурс».
- При **`?draft=uuid`**: загрузка состояния с сервера; при ошибке — новый UUID и снова список черновиков.
- **Автосохранение:** после `readStepIntoState` при нажатии «Далее» или «Назад» выполняется тихий `PUT` черновика (даже если валидация шага не прошла — на диске остаётся последняя зафиксированная версия полей).
- **«Временное сохранение»:** явный `PUT` с кратким сообщением об успехе.
- **«Грязность» для защиты ухода:** сравнение текущего сериализованного `state` с последним успешным ответом `PUT`; при первом заходе без правок с шага 0 уход без модалки (кроме явного «Отмена» с подтверждением, если данных нет — простой confirm на главную).

<a id="readme-6b6"></a>
### 6b.6. Защита ухода со страницы мастера

Аналогично идее `installLeaveGuard` у карточки строки, но привязана к **черновику в `wizard_draft`**, а не к сохранению строк листа:

| Ситуация | Поведение |
|----------|-----------|
| Есть несохранённый на сервере прогресс (после правок без успешного PUT) | `beforeunload`; клики по внешним ссылкам и submit «Остановить» перехватываются в фазе `capture`; показывается модалка с кнопками **«Сохранить черновик и выйти»** (сначала `PUT`, затем переход), **«Выйти без сохранения»**, **«Остаться»**. |
| Кнопка «Отмена» мастера | При «грязном» состоянии — та же модалка с переходом на `/` вместо немедленного ухода. |
| Успешный `POST …/commit` | Перед редиректом включается флаг приостановки защиты, чтобы не сработал `beforeunload`. |

Модалка использует те же базовые классы оверлея, что и уход с карточки строки (`.spod-date-modal-overlay`, `.spod-leave-modal-dialog`).

<a id="readme-6b7"></a>
### 6b.7. Даты и календарь

Поля дат из `editor_textareas` (`input_type` `date` / `datepicker`, `date_picker` или `storage_format` с `YYYY`, `MM`, `DD`) в мастере и на карточке строки рендерятся через **`spod_date_picker.js`** (`SpodDatePicker.buildShell`): скрытое значение `YYYY-MM-DD`, кнопка отображения, иконка календаря, общее модальное окно выбора даты. В модалке год задаётся в диапазоне **1000–4000**; рядом с быстрыми кнопками «Начало» и «Конец» есть **«4000-01-01»** для типовой даты «без срока» (как во входных выгрузках). Подсказка может относиться к **плоской** колонке листа или к пути внутри JSON-колонки — через **`json_path`** в том же блоке `hints` (например, колонка `REWARD_ADD_DATA`, путь `tagEndDT`). Подключение: в шаблонах **`wizard_new_contest.html`** и **`row_detail.html`** скрипт `spod_date_picker.js` идёт **синхронно** перед основным скриптом с `defer`. Общая функция **`SpodDatePicker.hintIsDate`** исключает подсказки-календари из ветки «textarea по min_rows» в `row_editor.js` и `wizard_contest.js`.

<a id="readme-6b8"></a>
### 6b.8. Commit, коды наград и турниров

| Тема | Поведение |
|------|-----------|
| **HTTP** | Успех: **303** + `Location` на карточку созданного CONTEST-DATA; клиент выполняет переход **после** завершения обработки ответа (`setTimeout` + `location.assign`), чтобы не прерывать `fetch` и не показывать ложную «Ошибку сети». Ошибка валидации: **400**, тело JSON с полем `detail`. Прочая ошибка сервера: **500**, тело JSON с `detail` (не HTML-страница ошибки). |
| **`REWARD_CODE`** | В шаге REWARD-LINK код собирается как `r_<CONTEST_CODE>` + необязательный суффикс; при **нескольких** связях суффикс обязателен и не должен повторяться. Проверка на сервере: `wizard_contest.validate_payload`. |
| **`TOURNAMENT_CODE`** | Формат `t_<CONTEST_CODE>_<dddd>` — после префикса **ровно четыре цифры**. Проверка на сервере: `validate_payload`. |

<a id="readme-6a1"></a>
### 6a.1. Диалог «Несохранённые изменения» (только при «грязной» форме)

Срабатывает, если текущее состояние полей (плоские + собранный JSON) **отличается** от состояния при загрузке страницы (серверный снимок в `bootstrap.__initialCanonical`).

| Действие пользователя | Поведение |
|------------------------|-----------|
| Любая **внешняя** ссылка на другой путь (в т.ч. другой лист, главная, связь в контенте), кроме исключений ниже | Клик перехватывается; открывается модалка; после выбора выполняется отложенный переход (`location.href` = URL ссылки). |
| **«Шаг назад»** (`#spod-trail-back`) | Модалка; при подтверждении — логика как в `trail_nav.js`: из `sessionStorage` (`spod_edit_trail`) снимается предыдущий шаг, переход на его `href` или на список текущего листа, если цепочка пуста. |
| **Крошки** с атрибутом `data-trail-go` | Модалка; при подтверждении — обрезка цепочки до выбранного индекса и переход на соответствующий URL. |
| **«Назад к списку»** в доке (это ссылка на `/sheet/...`) | Модалка (уход со страницы строки с правками). |
| **«Сохранить в базу»** / **«Сохранить черновик»** / **«Отменить правку»** | Модалка ухода **не** показывается: сохранение в базу — через `fetch` … `/save` и редирект (перед редиректом вызывается `__spodSuspendLeaveForNavigation`); сохранение черновика — `PUT /draft`; отмена — `location.reload()` с предварительным `leaveGuardSuspended`, без системного диалога Safari/Chrome. |
| **«Остановить»** (submit формы в шапке) | При грязной форме submit отменяется, показывается модалка; при «Сохранить и выйти» — сначала сохранение, затем повторная отправка формы остановки; при «Выйти без сохранения» — отправка формы без сохранения в БД (возможен встроенный `confirm` в шаблоне — отдельно от модалки правок). |
| Клик по **фону** модалки ухода | Закрытие модалки, переход **отменяется** (как у модалки календаря дат). |
| **Закрытие вкладки**, **перезагрузка F5**, уход по адресной строке | Браузерный `beforeunload` (текст диалога задаёт браузер; это отдельный слой от HTML-модалки). |

**Кнопки модалки ухода**

| Кнопка | Действие |
|--------|----------|
| **Сохранить и выйти** | POST `/sheet/{code}/row/{id}/save` с текущим телом формы; при успехе обновляется «чистый» снимок и выполняется запланированный уход (ссылка, шаг назад, остановка и т.д.). |
| **Сохранить черновик и выйти** | Сохраняет запись `row_edit_draft` со статусом **EDIT** и выполняет запланированный уход; если есть неподтверждённые поля (без **✓**), перед сохранением показывается доп. диалог с выбором: сохранить только подтверждённые / не сохранять и выйти / продолжить правку. |
| **Выйти без сохранения** | Запланированный уход **без** записи в БД; перед навигацией включается приостановка защиты, чтобы **не** появилось второе системное окно поверх уже принятого решения. |
| **Остаться** | Закрытие модалки, остаёмся на странице редактирования, правки в полях сохраняются в памяти браузера. |

**Технические детали (для сопровождения):** перехват кликов — `document` + `capture: true`; игнорируются модификаторы Ctrl/Meta/Shift/Alt для открытия в новой вкладке, `target="_blank"`, `download`, якорь на тот же документ без смены пути (`#…`), пустой `href`. Классы оверлея: `spod-date-modal-overlay`, `spod-leave-modal-overlay`, `spod-date-modal-overlay--closed`; у оверлея ухода повышенный `z-index` в CSS, чтобы перекрывать модалку выбора даты.

**Ограничение:** добавление новых элементов в массив через форму не поддерживается — только правка существующих «листьев»; расширение структуры — в режиме «Сырой JSON».

<a id="readme-7"></a>
## 7. Запуск

Из корня **этого** репозитория:

```bash
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python main.py
# или: ./run.sh
```

**Важно:** зависимости (`uvicorn`, `fastapi`, …) ставятся **только** из `requirements.txt` в **это** виртуальное окружение (`.venv` в корне репозитория). Запуск через «чужой» Python без `pip install -r requirements.txt` даст ошибку `ModuleNotFoundError: uvicorn`.

**Если в IDE выбран Python не из `.venv` этой панели** (например, из другого проекта), не запускайте `main.py` этим интерпретатором. Варианты:

- в терминале из корня репозитория: **`./run.sh`** — вызывает `.venv/bin/python main.py`;
- либо в Cursor: **Python: Select Interpreter** → указать `.venv/bin/python`, затем **Run** на `main.py`.

Откройте в браузере адрес из консоли (по умолчанию `http://127.0.0.1:8765/`).

**Старт приложения и БД:** при поднятии FastAPI выполняются `init_schema`, при необходимости добавление `sheet.headers_json`, миграция удаления устаревшей `data_row`, создание/синхронизация таблиц `spod_sheet_*` с текущими CSV (`sheet_storage.rebuild_all_sheet_tables_from_config`). Если набор колонок листа изменился (в т.ч. из‑за `json_columns`), таблица пересоздаётся и **сразу** подгружается CSV этого листа (`ingest.reimport_sheet`), иначе при непустом реестре `sheet` списки остались бы пустыми. Если таблица `sheet` пуста — автоматический полный импорт из `IN/SPOD` и пересчёт консистентности.

Проверки: `python -m unittest discover -s src/Tests -p "test_*.py"`

**Остановка сервера:** кнопка **«Остановить»** в шапке (POST `/admin/stop`) или в терминале **Ctrl+C**; либо `lsof -i :8765` и `kill <PID>`.
