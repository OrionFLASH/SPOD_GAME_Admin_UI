# SPOD Game Admin UI (панель турниров и конкурсов)

Локальная панель администратора для просмотра и правки выгрузок турниров SPOD (CSV с разделителем `;`), с хранением в SQLite и упрощёнными проверками связей между листами. Запуск одной командой `python main.py` (встроенный Uvicorn), без Docker и отдельных сервисов.

**Репозиторий:** [SPOD_GAME_Admin_UI](https://github.com/OrionFLASH/SPOD_GAME_Admin_UI) — код панели в **корне** этого каталога (ранее каталог `spod_tournament_admin/` в репозитории SPOD_PROM). История коммитов, относящаяся к панели, перенесена из старого репозитория командой `git subtree split -P spod_tournament_admin` (сообщения и содержимое коммитов сохранены).

**Локальная разработка:** откройте в Cursor корень этого репозитория (`SPOD_GAME_Admin_UI`), чтобы подсказки и поиск кода относились только к панели.

**История агента Cursor (локально):** в каталог данных Cursor для этого workspace скопированы архивы диалогов из старых папок проекта (`SPOD-PROM` и при открытии подпапки — `spod-tournament-admin`):  
`~/.cursor/projects/Users-orionflash-Documents-MyProject-SPOD-GAME-Admin-UI/agent-transcripts/`  
(файлы `*.jsonl` в подкаталогах по UUID). После копирования имеет смысл **полностью закрыть Cursor и открыть проект заново** — отображение старых чатов в боковой панели зависит от версии Cursor и не документировано как официальная функция; при отсутствии чатов в UI файлы всё равно можно открыть вручную для справки. В Git эти файлы не входят.

**Пути:** `IN/`, `OUT/`, `log/` — относительно корня этого репозитория. Свой `requirements.txt` и виртуальное окружение `.venv/` в корне.

**`.gitignore`:** в репозитории не коммитятся `OUT/`, логи, `.venv/`; входные CSV в `IN/**/*.csv` **разрешены** к коммиту (см. исключение в `.gitignore`).

## 1. Формулировка задачи и техническое задание

1. Импорт шести листов из `IN/SPOD` в базу данных SQLite.
2. Просмотр списка строк по каждому листу с поиском (включая подтянутые названия), колонками «Ключ», «Название», «Связи» и индикатором консистентности.
3. Просмотр и редактирование одной строки: плоские поля в **сетке** с поиском; JSON-колонки — **развёртка в отдельные параметры** (строка, число, логический, null) плюс режим **«Сырой JSON»**; боковая навигация по разделам; признак изменённых полей (предыдущее значение рядом); нижняя панель **«Сохранить в базу»** / **«Отменить правку»** (запись в БД только по сохранению; отмена — перезагрузка страницы с сервера); переходы по **связям** между строками разных листов с цепочкой возврата и подменю в колонке «Разделы». **Защита от потери правок:** при несохранённых отличиях формы от загруженных данных — перехват ухода на другой адрес, шаг «назад» по цепочке, переход по крошкам, ссылка «Назад к списку», отправка формы **«Остановить»** с показом **собственного** диалога (сохранить и уйти / выйти без сохранения / остаться); при закрытии вкладки или обновлении страницы — стандартное предупреждение браузера (`beforeunload`). Кнопки дока «Сохранить в базу» и «Отменить правку» **не** должны дублировать этот диалог; после выбора в модалке ухода не должно появляться **второго** системного окна из‑за `beforeunload`.
4. Сохранение **новой версии** строки в БД (старая версия остаётся в истории, актуальна только последняя) с пересчётом проверок; опционально запрет сохранения при ошибках (`consistency.mode: strict`).
5. Экспорт листа обратно в CSV (`;`, UTF-8).
6. Переимпорт всех CSV из папки (сброс данных в БД).
7. Светлый интерфейс в духе «liquid glass» (полупрозрачные панели, размытие фона).
8. Кнопка **«Остановить»** в шапке на всех экранах: POST `/admin/stop` — завершение процесса панели (и дочерних процессов на POSIX через `pgrep -P`).
9. **Мастер создания конкурса** по шагам (CONTEST-DATA → … → предпросмотр) с промежуточным сохранением черновика в БД, возобновлением по списку или `?draft=uuid`, финальной атомарной вставкой строк в листы и защитой ухода со страницы при несохранённом относительно сервера состоянии.

## 2. Описание решения

- **Стек:** Python 3, FastAPI, Uvicorn, Jinja2, стандартный `sqlite3`, статика и шаблоны в `src/`.
- **Конфигурация:** `config.json` — пути, порт, список листов `sheets` (импорт и UI), справочник соответствий `sheet_bindings` (код листа ↔ название ↔ CSV), пояснение модели БД `database_model`, перечисления для редактора `field_enums` (по листам с массивом `rules`), подсказки высоты textarea `editor_textareas` (по листам с массивом `hints`), порог длинного текста `editor_long_text_threshold`, режим консистентности.
- **Развёртка настроек редактора:** `src/editor_config.py` — превращает сгруппированные `field_enums` / `editor_textareas` / `editor_field_ui` в плоские списки для `row_editor.js`.
- **Проверка конфига:** `src/config_validate.py` — при старте сравнивает `sheet_bindings` с `sheets` (код и имя файла) и пишет предупреждения в лог при расхождении.
- **БД:** файл `OUT/DB/tournament_admin.sqlite`; реестр листов **`sheet`** (в т.ч. `headers_json` — порядок колонок CSV); для каждого листа — отдельная таблица **`spod_sheet_<КОД>`** (`src/sheet_storage.py`, `src/db.py`): колонки как в CSV, JSON-колонки как TEXT с полным JSON, плюс денормализованные листья **`j__<имя_JSON_колонки>__<путь>`** для SQL-запросов по отдельным полям. Версионирование: `is_current`, `replaces_row_id`, `sort_key`, `row_index`, `sheet_id` — как раньше по смыслу, но на физической таблице листа. Повторное сохранение без изменений — HTTP 400. **`wizard_draft`** не участвует в импорте CSV. При обнаружении устаревшей таблицы **`data_row`** она удаляется, данные листов нужно снова загрузить импортом (автоматически при пустом `sheet`).
- **Импорт:** `src/ingest.py` читает CSV, создаёт/пересоздаёт таблицу листа и вставляет строки с денормализацией JSON.
- **JSON SPOD:** `src/spod_json.py` — нормализация тройных кавычек и попытка `json.loads` для отображения в редакторе; на клиенте ту же логику повторяет **`SpodJsonEditor.tryParseSpodJsonCell`** в `row_editor.js` (сравнение «грязного» JSON и мастер при копии из БД).
- **Проверки:** `src/consistency.py` — множества кодов конкурсов/наград, пары `(CONTEST_CODE, GROUP_CODE)`, ссылки REWARD-LINK, INDICATOR, TOURNAMENT-SCHEDULE.
- **Связи на карточке строки:** `src/relations.py` — выборки связанных фрагментов по кодам; в шаблоне — ссылки на `/sheet/{code}/row/{id}` с подписью `preview` (короткий текст **«: значение»**, без префикса `KEY=`), якоря `rel-card-*` для прокрутки к карточке.
- **Навигация по связям и возврат:** `src/static/trail_nav.js` — при переходе по связи в `sessionStorage` сохраняется цепочка страниц; со списка листа в строку добавляется `?from_list=1` (после загрузки параметр убирается из адреса), чтобы первым шагом «назад» был список этого листа; панель над формой: крошки, «Шаг назад», «К списку листа», «На главную»; сброс цепочки при открытии `/` и при ссылке «Панель турниров».
- **Остановка сервера из UI:** `src/server_stop.py` + маршрут `POST /admin/stop` — после ответа браузеру планируется `SIGTERM` дочерним процессам текущего PID (если есть), затем процессу панели; на POSIX через ~2.5 с при необходимости повторно отправляется `SIGKILL` самому процессу. **Без пароля:** рассчитано на локальный `127.0.0.1`; не выставляйте панель в открытую сеть без прокси с авторизацией.
- **Экспорт:** `src/export_csv.py` — восстановление CSV по заголовкам из первой строки данных листа.
- **Мастер «Создать конкурс»:** кнопка на главной → `GET /wizard/new-contest` (`wizard_new_contest.html`, `wizard_contest.js`); подключается **`row_editor.js`** для общего UI JSON-колонок (`window.SpodJsonEditor`) и бейджей ограничений (`window.SpodFieldUiSignals`). Порядок шагов: CONTEST-DATA → … → предпросмотр; **`POST /wizard/new-contest/commit`** — вставка строк в таблицы листов `spod_sheet_*`; ответ **303** с `Location` на карточку конкурса; на клиенте переход отложен (`setTimeout` + `location.assign`), чтобы не ловить ложный **AbortError** у `fetch`; ошибки разбираются из `response.text()` (JSON с `detail` или текст). **Коды:** `REWARD_CODE` в шаге REWARD-LINK — префикс `r_<CONTEST_CODE>` и поле суффикса (при нескольких связях суффикс обязателен и уникален); `TOURNAMENT_CODE` — префикс `t_<CONTEST_CODE>_` и **ровно четыре цифры**; проверки дублируются в `wizard_contest.validate_payload`. **Черновик:** `wizard_draft`, `PUT`/`GET`/`DELETE` draft API; при commit передаётся `draft_uuid` — черновик удаляется в транзакции. **Уход:** модалка и `beforeunload` при несохранённом относительно сервера состоянии.
- **Редактор строки (клиент):** `row_editor.js` — карточка строки из `#row-editor-bootstrap`; плоские поля и JSON-колонки с режимами **«По полям»** / **«Сырой JSON»**; пустой JSON и `{}` дополняются шаблоном из `field_enums` и обязательных `json_path` в `editor_field_ui` (`mergeDeclaredJsonTemplate`); `field_enums` — select и «Задать своё…»; `editor_textareas` — высота textarea и **поля дат** (в т.ч. с `json_path` внутри JSON-колонки); порог длинного текста; ограничения полей — бейджи **«Обязательно»** / **«Не пусто»**; экспорт **`SpodJsonEditor`** / **`SpodFieldUiSignals`** для мастера. Для **`REWARD` / `REWARD_ADD_DATA`** по значению **`REWARD_TYPE`** (из плоских полей или `flat` в мастере) в режиме «По полям» показываются только ключи, допустимые для типа по каталогу `Docs/JSON/SPOD_INPUT_DATA_CATALOG.md`; непустые «чужие» ключи из данных остаются видимыми; в шаблон не подставляются пути, не допускаемые для выбранного типа. **Разбор JSON в мастере:** перед развёрткой вызывается **`tryParseSpodJsonCell`** (та же нормализация SPOD, что и `spod_json.try_parse_cell`), чтобы при загрузке копии конкурса из БД работал режим «По полям» для колонок вроде `CONTEST_FEATURE`. **Защита ухода (`installLeaveGuard`):** перехват ссылок, модалка, `beforeunload`; флаг `leaveGuardSuspended`; `spod_edit_trail`; стили `.spod-leave-modal-overlay` в `app.css`.

Точка входа: `main.py` или **`run.sh`** поднимают Uvicorn на `host`/`port` из конфига (по умолчанию `http://127.0.0.1:8765/`).

## 4. Переменные и ключи `config.json`

| Ключ | Назначение |
|------|------------|
| `paths.input_spod` | Каталог с CSV |
| `paths.output_db_dir` | Каталог для SQLite |
| `paths.logs` | Каталог логов |
| `database.filename` | Имя файла БД |
| `server.host`, `server.port` | Адрес HTTP |
| `logging.level`, `logging.base_name` | Уровень и префикс имени файла лога |
| `consistency.mode` | `warn` — сохранять и помечать ошибки; `strict` — откат при ошибках на сохраняемой строке; `soft` зарезервировано как синоним мягкого поведения |
| `database_model` | Справка: `sheet`, префикс таблиц `spod_sheet_*`, ограничения FK между листами; массив **`logical_relationships_ru`** — зафиксированная логика связей сущностей (дублирует доменную модель для людей и для согласования с `consistency.py`) |
| `sheet_bindings[]` | Для каждого листа: `code`, `title`, `csv_file` — должно совпадать с `sheets` (проверка при старте) |
| `editor_long_text_threshold` | Минимальная длина строки (символы), после которой в JSON-редакторе показывается textarea вместо однострочного поля |
| `field_enums[]` | Блоки по листу: `sheet_code` + массив `rules` (`column`, `options`, `allow_custom`, опционально `json_path`). Для **ограниченного набора** строк (справочники, коды перечислений). Свободные даты в формате ISO задаются через **`editor_textareas`** с типом даты, а не длинным списком дат в `options`. |
| `editor_textareas[]` | Блоки по листу: `sheet_code` + массив `hints` (`column`, `min_rows`, `max_rows`, опционально **`json_path`** для листа внутри JSON-колонки, напр. `REWARD_ADD_DATA` → `tagEndDT`). Для дат: `input_type` `date` / `datepicker` или `date_picker`, либо `storage_format` с `YYYY`/`MM`/`DD` — календарь `spod_date_picker.js` на карточке строки и в мастере для **плоских** колонок; примеры в конфиге: `PLAN_PERIOD_START_DT`, `PLAN_PERIOD_END_DT` (TOURNAMENT-SCHEDULE), дата в `REWARD_ADD_DATA.tagEndDT`. |
| `editor_field_ui[]` | Блоки по листу: `sheet_code` + массив `rules` — для каждого редактируемого поля: `column`; опционально `json_path` (как в `field_enums`) для листа внутри JSON-колонки; `label` — подпись в форме (по умолчанию на клиенте — имя колонки или путь); `description` — пояснение; `show_description` — если `true`, текст `description` выводится сразу под подписью поля мелким приглушённым шрифтом; `required` — рядом с подписью бейдж **«Обязательно»** (подсказка по наведению); `allows_empty` — при `false` бейдж **«Не пусто»** (пустое значение недопустимо; независимо от `required`). Флаги для плоских колонок задаёт скрипт `scripts/generate_editor_field_ui.py` по анализу CSV. |
| `sheets[].code` | Внутренний код листа (URL); источник правды для импорта |
| `sheets[].title` | Заголовок в UI |
| `sheets[].file` | Имя файла в `IN/SPOD` (должно совпадать с `sheet_bindings[].csv_file`) |
| `sheets[].json_columns` | Колонки, выносимые в JSON-редактор |
| `sheets[].primary_key_column` | Колонка для превью в таблице списка |

**Пример:** после правок JSON перезапуск не обязателен; смена `config.json` требует перезапуска. При смене имени CSV обновите и `sheets[].file`, и `sheet_bindings[].csv_file`.

### 4.1. Логическая модель связей между листами

Полный текст в **`config.json` → `database_model.logical_relationships_ru`**. Кратко:

| Откуда | Куда | Ключ | Кардинальность | Смысл |
|--------|------|------|------------------|--------|
| **CONTEST-DATA** | — | `CONTEST_CODE` | уникален среди актуальных строк | Справочник конкурсов |
| **GROUP** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много групп |
| **REWARD-LINK** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много связей |
| **REWARD-LINK** | GROUP | `(CONTEST_CODE, GROUP_CODE)` | **N:1** | Пара должна существовать в GROUP |
| **REWARD-LINK** | **REWARD** | `REWARD_CODE` | **N:1** | Много строк связи могут ссылаться на **одну** строку награды с тем же кодом (в быту говорят «награда одна — ссылок много»; не путать с дублированием строк в REWARD) |
| **REWARD** | CONTEST-DATA | транзитивно через REWARD-LINK | согласование | Все `REWARD-LINK` с данным `REWARD_CODE` должны иметь **один и тот же** `CONTEST_CODE`; иначе награда «приписана» к разным конкурсам |
| **INDICATOR** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много показателей |
| **TOURNAMENT-SCHEDULE** | CONTEST-DATA | `CONTEST_CODE` | **N:1** | Одному конкурсу — много строк расписания |

Проверки: `consistency.py` — ссылки на существующие коды, уникальность `CONTEST_CODE` / `REWARD_CODE` в справочниках, отсутствие «висячих» наград без `REWARD-LINK`, запрет разных `CONTEST_CODE` для одного `REWARD_CODE` в связях.

## 5. Основные функции модулей

## 6b. Мастер «Создать конкурс» (подробно)

### 6b.1. Назначение и порядок шагов

Мастер ведёт пользователя по цепочке без записи в таблицы листов до последнего шага. После подтверждения на экране предпросмотра сервер выполняет `wizard_contest.commit_wizard`: вставка строк в `spod_sheet_*` (CONTEST-DATA, GROUP, …), затем `consistency.run_all_checks`; в режиме `strict` при ошибке на одной из вставленных строк выполняется откат транзакции.

| Шаг (индекс) | Содержание |
|----------------|------------|
| 0 | Плоские поля + JSON колонки листа **CONTEST-DATA** |
| 1 | Число строк **GROUP**, формы по каждой строке; `CONTEST_CODE` подставляется из шага 0 |
| 2 | **REWARD-LINK**: не меньше числа различных `GROUP_CODE`; связь с `CONTEST_CODE` |
| 3 | **REWARD**: по одному блоку на каждый уникальный `REWARD_CODE` из связей |
| 4 | **INDICATOR**: задаётся число строк, общий `CONTEST_CODE` |
| 5 | **TOURNAMENT-SCHEDULE**: число строк, общий `CONTEST_CODE` |
| 6 | Предпросмотр JSON и кнопка «Подтвердить создание в базе» |

Валидация обязательных полей и запрета пустых значений на шагах 0–5 выполняется в браузере по правилам `editor_field_ui` (`required`, `allows_empty`); у подписей полей отображаются текстовые бейджи **«Обязательно»** и **«Не пусто»** с пояснением в `title`.

### 6b.2. Таблица `wizard_draft`

Создаётся при старте приложения (`ensure_wizard_draft_table`). Назначение — **промежуточное** состояние мастера, чтобы после сбоя или закрытия вкладки можно было продолжить.

| Колонка | Назначение |
|---------|------------|
| `id` | Суррогатный ключ |
| `draft_uuid` | Уникальный идентификатор сессии (UUID с клиента) |
| `step_index` | Текущий шаг (0…6) |
| `status` | Для черновиков мастера используется значение **`EDIT`** |
| `state_json` | JSON-объект полного состояния мастера (см. ниже) |
| `contest_code_preview` | Копия `CONTEST_CODE` из состояния для отображения в списке (может быть пусто) |
| `updated_at` | Метка времени UTC (`YYYY-MM-DDTHH:MM:SSZ`) |

Черновик **не** дублирует бизнес-данные в таблицах листов до `commit`.

### 6b.3. Формат `state` для `PUT /wizard/new-contest/draft`

Тело запроса:

```json
{
  "draft_uuid": "строка-uuid",
  "step_index": 0,
  "state": { ... }
}
```

Объект **`state`** (внутри `state_json` на сервере) содержит поля:

| Поле | Тип смысла |
|------|------------|
| `stepIndex` | Номер шага (дублирует `step_index` запроса; при расхождении приоритет у поля запроса при записи `step_index` в таблицу) |
| `contest` | `{ "cells": { "COL": "значение", ... } }` |
| `groups`, `reward_links`, `rewards`, `indicators`, `schedules` | Массивы `{ "cells": { ... } }` |
| `groupCount`, `linkCount`, `indicatorCount`, `scheduleCount` | Числа для UI «сформировать N строк» |

### 6b.4. API черновика (кратко)

- **`GET /wizard/new-contest/drafts`** — массив объектов `{ draft_uuid, step_index, contest_code_preview, updated_at }`.
- **`GET /wizard/new-contest/draft/{uuid}`** — 404, если черновик не найден.
- **`PUT /wizard/new-contest/draft`** — upsert по `draft_uuid` (SQLite `ON CONFLICT DO UPDATE`).
- **`DELETE /wizard/new-contest/draft/{uuid}`** — удаление (кнопка «Удалить» у строки в списке на странице мастера).

### 6b.5. Поведение клиента (`wizard_contest.js`)

- При открытии `/wizard/new-contest` без параметров: запрашивается список черновиков; при наличии — панель с кнопками «Продолжить» (переход на `?draft=uuid`), «Удалить», «Начать новый конкурс».
- При **`?draft=uuid`**: загрузка состояния с сервера; при ошибке — новый UUID и снова список черновиков.
- **Автосохранение:** после `readStepIntoState` при нажатии «Далее» или «Назад» выполняется тихий `PUT` черновика (даже если валидация шага не прошла — на диске остаётся последняя зафиксированная версия полей).
- **«Временное сохранение»:** явный `PUT` с кратким сообщением об успехе.
- **«Грязность» для защиты ухода:** сравнение текущего сериализованного `state` с последним успешным ответом `PUT`; при первом заходе без правок с шага 0 уход без модалки (кроме явного «Отмена» с подтверждением, если данных нет — простой confirm на главную).

### 6b.6. Защита ухода со страницы мастера

Аналогично идее `installLeaveGuard` у карточки строки, но привязана к **черновику в `wizard_draft`**, а не к сохранению строк листа:

| Ситуация | Поведение |
|----------|-----------|
| Есть несохранённый на сервере прогресс (после правок без успешного PUT) | `beforeunload`; клики по внешним ссылкам и submit «Остановить» перехватываются в фазе `capture`; показывается модалка с кнопками **«Сохранить черновик и выйти»** (сначала `PUT`, затем переход), **«Выйти без сохранения»**, **«Остаться»**. |
| Кнопка «Отмена» мастера | При «грязном» состоянии — та же модалка с переходом на `/` вместо немедленного ухода. |
| Успешный `POST …/commit` | Перед редиректом включается флаг приостановки защиты, чтобы не сработал `beforeunload`. |

Модалка использует те же базовые классы оверлея, что и уход с карточки строки (`.spod-date-modal-overlay`, `.spod-leave-modal-dialog`).

### 6b.7. Даты и календарь

Поля дат из `editor_textareas` (`input_type` `date` / `datepicker`, `date_picker` или `storage_format` с `YYYY`, `MM`, `DD`) в мастере и на карточке строки рендерятся через **`spod_date_picker.js`** (`SpodDatePicker.buildShell`): скрытое значение `YYYY-MM-DD`, кнопка отображения, иконка календаря, общее модальное окно выбора даты. В модалке год задаётся в диапазоне **1000–4000**; рядом с быстрыми кнопками «Начало» и «Конец» есть **«4000-01-01»** для типовой даты «без срока» (как во входных выгрузках). Подсказка может относиться к **плоской** колонке листа или к пути внутри JSON-колонки — через **`json_path`** в том же блоке `hints` (например, колонка `REWARD_ADD_DATA`, путь `tagEndDT`). Подключение: в шаблонах **`wizard_new_contest.html`** и **`row_detail.html`** скрипт `spod_date_picker.js` идёт **синхронно** перед основным скриптом с `defer`. Общая функция **`SpodDatePicker.hintIsDate`** исключает подсказки-календари из ветки «textarea по min_rows» в `row_editor.js` и `wizard_contest.js`.

### 6b.8. Commit, коды наград и турниров

| Тема | Поведение |
|------|-----------|
| **HTTP** | Успех: **303** + `Location` на карточку созданного CONTEST-DATA; клиент выполняет переход **после** завершения обработки ответа (`setTimeout` + `location.assign`), чтобы не прерывать `fetch` и не показывать ложную «Ошибку сети». Ошибка валидации: **400**, тело JSON с полем `detail`. Прочая ошибка сервера: **500**, тело JSON с `detail` (не HTML-страница ошибки). |
| **`REWARD_CODE`** | В шаге REWARD-LINK код собирается как `r_<CONTEST_CODE>` + необязательный суффикс; при **нескольких** связях суффикс обязателен и не должен повторяться. Проверка на сервере: `wizard_contest.validate_payload`. |
| **`TOURNAMENT_CODE`** | Формат `t_<CONTEST_CODE>_<dddd>` — после префикса **ровно четыре цифры**. Проверка на сервере: `validate_payload`. |

| Панель над формой | Крошки по цепочке переходов, кнопки «Шаг назад», «К списку листа», «На главную» |
| Список строк листа | Ссылка «Просмотр» ведёт на `/sheet/{code}/row/{id}?from_list=1` |
| «Поля таблицы» | Сетка `auto-fill`, фильтр по имени колонки; для колонок из `field_enums` — select и при необходимости поле «Задать своё…»; при наличии `editor_field_ui` — бейджи «Обязательно» / «Не пусто» |
| Каждая JSON-колонка | Карточка: фильтр по пути параметра, переключатель «По полям» / «Сырой JSON», сборка в компактный JSON при сохранении; отдельные параметры могут быть списками по `json_path` в конфиге |
| Нижняя панель | Фиксированная: баннер при несохранённых правках; «Сохранить в базу» и «Отменить правку» активны только при отличии формы от загруженных данных; «Назад к списку» — переход на список текущего листа и сброс цепочки `trail_nav.js` для этого листа |
| Изменённые поля | У скаляров — строка «Было: …» под полем; у JSON-блока — панель с прежним значением (при необходимости отформатированным) |

### 6a.1. Диалог «Несохранённые изменения» (только при «грязной» форме)

Срабатывает, если текущее состояние полей (плоские + собранный JSON) **отличается** от состояния при загрузке страницы (серверный снимок в `bootstrap.__initialCanonical`).

| Действие пользователя | Поведение |
|------------------------|-----------|
| Любая **внешняя** ссылка на другой путь (в т.ч. другой лист, главная, связь в контенте), кроме исключений ниже | Клик перехватывается; открывается модалка; после выбора выполняется отложенный переход (`location.href` = URL ссылки). |
| **«Шаг назад»** (`#spod-trail-back`) | Модалка; при подтверждении — логика как в `trail_nav.js`: из `sessionStorage` (`spod_edit_trail`) снимается предыдущий шаг, переход на его `href` или на список текущего листа, если цепочка пуста. |
| **Крошки** с атрибутом `data-trail-go` | Модалка; при подтверждении — обрезка цепочки до выбранного индекса и переход на соответствующий URL. |
| **«Назад к списку»** в доке (это ссылка на `/sheet/...`) | Модалка (уход со страницы строки с правками). |
| **«Сохранить в базу»** / **«Отменить правку»** | Модалка ухода **не** показывается: сохранение через `fetch` … `/save` и редирект (перед редиректом вызывается `__spodSuspendLeaveForNavigation`, чтобы не сработал лишний `beforeunload`); отмена — `location.reload()` с предварительным `leaveGuardSuspended`, без системного диалога Safari/Chrome. |
| **«Остановить»** (submit формы в шапке) | При грязной форме submit отменяется, показывается модалка; при «Сохранить и выйти» — сначала сохранение, затем повторная отправка формы остановки; при «Выйти без сохранения» — отправка формы без сохранения в БД (возможен встроенный `confirm` в шаблоне — отдельно от модалки правок). |
| Клик по **фону** модалки ухода | Закрытие модалки, переход **отменяется** (как у модалки календаря дат). |
| **Закрытие вкладки**, **перезагрузка F5**, уход по адресной строке | Браузерный `beforeunload` (текст диалога задаёт браузер; это отдельный слой от HTML-модалки). |

**Кнопки модалки ухода**

| Кнопка | Действие |
|--------|----------|
| **Сохранить и выйти** | POST `/sheet/{code}/row/{id}/save` с текущим телом формы; при успехе обновляется «чистый» снимок и выполняется запланированный уход (ссылка, шаг назад, остановка и т.д.). |
| **Выйти без сохранения** | Запланированный уход **без** записи в БД; перед навигацией включается приостановка защиты, чтобы **не** появилось второе системное окно поверх уже принятого решения. |
| **Остаться** | Закрытие модалки, остаёмся на странице редактирования, правки в полях сохраняются в памяти браузера. |

**Технические детали (для сопровождения):** перехват кликов — `document` + `capture: true`; игнорируются модификаторы Ctrl/Meta/Shift/Alt для открытия в новой вкладке, `target="_blank"`, `download`, якорь на тот же документ без смены пути (`#…`), пустой `href`. Классы оверлея: `spod-date-modal-overlay`, `spod-leave-modal-overlay`, `spod-date-modal-overlay--closed`; у оверлея ухода повышенный `z-index` в CSS, чтобы перекрывать модалку выбора даты.

**Ограничение:** добавление новых элементов в массив через форму не поддерживается — только правка существующих «листьев»; расширение структуры — в режиме «Сырой JSON».

## 7. Запуск

Из корня **этого** репозитория:

```bash
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
python main.py
# или: ./run.sh
```

**Важно:** зависимости (`uvicorn`, `fastapi`, …) ставятся **только** из `requirements.txt` в **это** виртуальное окружение (`.venv` в корне репозитория). Запуск через «чужой» Python без `pip install -r requirements.txt` даст ошибку `ModuleNotFoundError: uvicorn`.

**Если в IDE выбран Python не из `.venv` этой панели** (например, из другого проекта), не запускайте `main.py` этим интерпретатором. Варианты:

- в терминале из корня репозитория: **`./run.sh`** — вызывает `.venv/bin/python main.py`;
- либо в Cursor: **Python: Select Interpreter** → указать `.venv/bin/python`, затем **Run** на `main.py`.

Откройте в браузере адрес из консоли (по умолчанию `http://127.0.0.1:8765/`).

**Старт приложения и БД:** при поднятии FastAPI выполняются `init_schema`, при необходимости добавление `sheet.headers_json`, миграция удаления устаревшей `data_row`, создание/синхронизация таблиц `spod_sheet_*` с текущими CSV (`sheet_storage.rebuild_all_sheet_tables_from_config`). Если таблица `sheet` пуста — автоматический полный импорт из `IN/SPOD` и пересчёт консистентности.

Проверки: `python -m unittest discover -s src/Tests -p "test_*.py"`

**Остановка сервера:** кнопка **«Остановить»** в шапке (POST `/admin/stop`) или в терминале **Ctrl+C**; либо `lsof -i :8765` и `kill <PID>`.
